# 5. Accessibility

[[5. FED/5. Accessibility (a11y)/1. accessibility]] · [[6. System Design/6. Trade-offs]]

## Зачем в system design

Доступность — часть архитектуры, а не «добавка в конце». Нужно закладывать её в дизайн компонентов, маршрутизации и состояний.

## Интеграция в дизайн

### Компоненты

- Семантическая разметка по умолчанию
- ARIA только когда нативного HTML недостаточно
- Поддержка клавиатуры из коробки
- Управление фокусом (модалки, роутинг)

### Маршрутизация

- Skip links — «Перейти к контенту»
- Фокус при смене страницы — на заголовок или начало контента
- Объявление смены страницы для скринридеров (`aria-live`)

### Состояния и ошибки

- Ошибки — не только цвет (иконка, текст)
- Загрузка — `aria-busy`, `aria-live` для статуса
- Успех/ошибка — объявление через `aria-live="polite"`

### Дизайн-система

- Контраст, размеры, отступы — в токенах
- `prefers-reduced-motion` — в базовых стилях
- Фокус — видимое кольцо по умолчанию

## Trade-offs

- **Время разработки** — доступность с самого начала дешевле, чем доработка
- **Кастомные компоненты** — больше кода для ARIA и клавиатуры
- **Визуальный дизайн** — иногда нужны компромиссы (контраст, размер кнопок)

## Вопросы для обсуждения

### Как встроить a11y в процесс разработки?

**Ответ:** 1) С самого начала — в дизайн-системе и компонентах. 2) Чеклист в PR — семантика, клавиатура, ARIA при необходимости. 3) Линтеры — eslint-plugin-jsx-a11y в CI. 4) В Definition of Done — проверка клавиатурой. 5) Обучение команды — базовые принципы, когда привлекать эксперта. 6) Дизайн-ревью — контраст, размеры, не только цвет для состояний. Доступность как часть критериев приёмки, а не отдельная задача.

### Как тестировать доступность?

**Ответ:** 1) Автоматизация — axe-core, Lighthouse, pa11y в CI. 2) Ручная проверка клавиатурой — Tab, Enter, Escape, стрелки. 3) Скринридер — NVDA, VoiceOver (хотя бы основные сценарии). 4) Увеличение до 200% — читаемость. 5) prefers-reduced-motion — отключение анимаций. 6) Контраст — инструменты (WebAIM, Contrast Checker). Регулярные a11y-аудиты, не только перед релизом.

### Как приоритизировать исправления?

**Ответ:** 1) Критичные — блокируют использование (нет фокуса, не работает клавиатура, контент недоступен). 2) Серьёзные — затрудняют использование (плохой контраст, мелкие цели, отсутствие подписей). 3) Улучшения — удобство (порядок фокуса, избыточные объявления). Начать с критичных. Учитывать охват — исправления в shared-компонентах дают больше. WCAG уровни (A, AA, AAA) как ориентир.
