## Зачем на собеседовании

Умение разбивать систему на независимые части. Меньше связанности, проще тестировать и менять.

## Принципы

### Разделение ответственности (SoC)

- Один модуль — одна зона ответственности
- UI, бизнес-логика, данные — отдельные слои

### Низкая связанность (Loose coupling)

- Модули общаются через чёткие интерфейсы
- Изменение одного модуля не ломает другие

### Высокая связность (High cohesion)

- Внутри модуля — логически связанный код
- Всё, что относится к одной задаче, в одном месте

## Frontend: разбиение приложения

### По фичам (Feature-based)

```
src/
  features/
    auth/
    dashboard/
    settings/
```

Каждая фича — свои компоненты, хуки, API.

### По слоям (Layered)

```
src/
  components/
  hooks/
  services/
  store/
```

### Гибрид (FSD, Clean Architecture)

```
src/
  shared/       # переиспользуемое
  entities/     # бизнес-сущности
  features/     # фичи
  widgets/      # композиции
  pages/        # страницы
```

## Границы модулей

- **Публичный API** — что модуль экспортирует
- **Внутренняя реализация** — скрыта
- **Зависимости** — только вниз (features не зависят от pages)

## Вопросы для обсуждения

### Как разбить монолитное приложение?

**Ответ:** Постепенно, по границам. 1) Найти естественные границы (по доменам, фичам). 2) Выделить модули с чётким API — что экспортируют, что скрывают. 3) Запретить кросс-зависимости (features не импортируют друг друга). 4) Вынести общее в shared. 5) При необходимости — вынести модуль в отдельный сервис (strangler fig). Начать с самого независимого или самого болезненного модуля.

### Как организовать зависимости между модулями?

**Ответ:** Зависимости только «вниз»: pages → features → entities → shared. Никогда shared не зависит от features. Использовать dependency injection или контекст для слабой связанности. Публичный API модуля — минимальный набор экспортов. Внутренняя структура — приватная. ESLint-правила (import restrictions) для контроля зависимостей.

### Когда создавать новый модуль?

**Ответ:** При появлении отдельной зоны ответственности. Признаки: логика повторяется в нескольких местах, команда/фича растёт, код в одной папке становится трудно обозримым. Правило трёх — абстрагировать при третьем повторении. Не создавать модуль «на будущее» — только при реальной потребности.
