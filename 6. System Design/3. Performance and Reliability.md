## Performance

### Initial load

- **Critical path** — минимум блокирующих ресурсов
- **Code splitting** — загружать только нужный код
- **Lazy load** — изображения, тяжёлые компоненты
- **Preload** — критичные ресурсы (шрифты, hero-изображения)
- **SSR/SSG** — быстрый FCP для контента

### Subsequent interactions

- **Кэширование** — API, статика, service worker
- **Оптимистичные обновления** — UI до ответа сервера
- **Prefetch** — подгрузка следующей страницы
- **Виртуализация** — длинные списки

### Data fetching

- **Deduplication** — один запрос на несколько подписчиков
- **Stale-while-revalidate** — показывать кэш, обновлять в фоне
- **Pagination / infinite scroll** — не грузить всё
- **Background sync** — обновление при возврате в приложение

## Reliability / Resilience

### Обработка ошибок

- **Error boundaries** — изоляция падений в UI
- **Fallback UI** — запасной интерфейс при ошибке
- **Retry** — повтор запроса при сбое (с backoff)
- **Graceful degradation** — работа при частичном отказе

### Сетевые проблемы

- **Offline** — Service Worker, кэш, очередь запросов
- **Timeout** — ограничение времени ожидания
- **Circuit breaker** — отключение неработающего сервиса
- **Fallback data** — показывать кэш при недоступности API

### Устойчивый UX

- **Loading states** — скелетоны, спиннеры
- **Optimistic updates** — откат при ошибке
- **Toast/уведомления** — информирование об ошибках
- **Retry UI** — кнопка «Повторить» при сбое

## Вопросы для обсуждения

### Как обрабатывать падение API?

**Ответ:** 1) Error boundary — изолировать падение, показать fallback. 2) Retry с exponential backoff — 1s, 2s, 4s. 3) Fallback на кэш — показывать stale data. 4) Toast/уведомление пользователю с кнопкой «Повторить». 5) Мониторинг и алерты. 6) Circuit breaker — при серии ошибок временно не слать запросы, дать сервису восстановиться. Критичные действия (оплата) — явный retry UI, не молча падать.

### Как обеспечить работу при нестабильной сети?

**Ответ:** 1) Service Worker — кэш статики и API-ответов. 2) Stale-while-revalidate — показывать кэш, обновлять в фоне. 3) Очередь запросов при offline — отправить при восстановлении. 4) Timeout — не ждать бесконечно, показать ошибку. 5) Optimistic updates — обновить UI сразу, откатить при ошибке. 6) Индикатор статуса сети (online/offline). 7) Сообщение «Нет соединения» вместо пустого экрана.

### Как не блокировать UI при тяжёлых операциях?

**Ответ:** 1) Web Workers — вынести вычисления в отдельный поток. 2) Разбить на chunks — `setTimeout(0)` или `requestIdleCallback` между порциями. 3) `useTransition` (React) — помечать обновления как некритичные. 4) Виртуализация — не рендерить тысячи элементов. 5) Debounce/throttle для частых событий. 6) Показывать loading state — скелетон, спиннер. 7) Lazy load — загружать тяжёлые компоненты по требованию.
