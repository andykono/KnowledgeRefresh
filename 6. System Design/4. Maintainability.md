# 4. Maintainability

[[6. System Design/2. Modularity]] · [[6. System Design/6. Trade-offs]]

## Зачем на собеседовании

Код живёт годами. Важно показать, что вы пишете так, чтобы его было легко менять, тестировать и понимать.

## Чистый код

### Именование

- Понятные имена переменных, функций, модулей
- Единый стиль (camelCase, константы и т.п.)

### Функции и компоненты

- Одна задача на функцию/компонент
- Короткие, без глубокой вложенности
- Мало аргументов (объект при необходимости)

### DRY и абстракция

- Не дублировать логику
- Абстрагировать только при реальном повторении (правило трёх)

## Тестируемость

- **Изоляция** — логика отделена от UI
- **Чистые функции** — предсказуемый результат
- **Зависимости через параметры** — не хардкод, injection
- **Покрытие** — unit для логики, integration для потоков

```javascript
// Сложно тестировать
function process() {
  const data = fetchFromAPI();  // зависимость внутри
  return data.map(x => x * 2);
}

// Легко тестировать
function process(data) {
  return data.map(x => x * 2);
}
```

## Понятность

- **Комментарии** — зачем, а не что (код показывает что)
- **Структура** — предсказуемое расположение файлов
- **Документация** — README, API, архитектурные решения (ADR)

## Консистентность

- Единые паттерны в проекте
- ESLint, Prettier — автоматическое форматирование
- Code review — общие стандарты

## Вопросы для обсуждения

### Как сделать код проще для новых разработчиков?

**Ответ:** 1) Единая структура проекта — onboarding doc, где что искать. 2) Понятные имена — функции и переменные объясняют назначение. 3) README с запуском, архитектурой, ключевыми решениями. 4) ADR (Architecture Decision Records) — почему выбран подход. 5) Code review — передача знаний, единые стандарты. 6) Модульность — можно разбирать по частям. 7) Тесты как документация — показывают ожидаемое поведение.

### Как балансировать абстракцию и простоту?

**Ответ:** Абстрагировать при реальном повторении (правило трёх). Не создавать абстракцию «на будущее». YAGNI — You Aren't Gonna Need It. Если сомневаешься — оставить проще, рефакторить при появлении дублирования. Абстракция должна уменьшать сложность, а не добавлять. Плохой признак — абстракция используется в одном месте.

### Как организовать тесты?

**Ответ:** Пирамида: много unit (логика, утилиты), меньше integration (компоненты + API), мало E2E (критичные сценарии). Unit — быстрые, изолированные, тестируют одну вещь. Integration — React Testing Library, проверка взаимодействий. E2E — Playwright/Cypress для ключевых потоков. Логику выносить из UI — тестировать отдельно. Моки — только для внешних зависимостей (API, router).
