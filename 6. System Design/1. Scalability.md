## Зачем на собеседовании

Показать, что вы думаете о росте: пользователи, данные, функциональность. Важно уметь обсуждать горизонтальное/вертикальное масштабирование, узкие места, стратегии.

## User load

### Горизонтальное масштабирование

- Добавление серверов/инстансов вместо апгрейда одного
- Load balancer — распределение запросов
- Stateless backend — сессии в Redis/вне приложения

### Вертикальное

- Больше CPU/RAM на одном сервере
- Ограничено, но проще на старте

### Frontend

- CDN для статики (HTML, JS, CSS, изображения)
- Edge caching — контент ближе к пользователю
- Изоляция: один пользователь не должен влиять на других

## Data growth

- **Пагинация / cursor-based** — не грузить всё сразу
- **Виртуализация** — рендер только видимых элементов
- **Lazy loading** — подгрузка по мере необходимости
- **Database** — индексы, шардинг, репликация
- **Кэширование** — Redis, in-memory, CDN

## Feature growth

- **Модульность** — независимые модули/микросервисы [[6. System Design/2. Modularity]]
- **Code splitting** — загрузка кода по маршрутам/фичам
- **Feature flags** — включение фич без деплоя
- **Монорепо / полирепо** — организация кода при росте команд

## Вопросы для обсуждения

### Что будет узким местом при 10x пользователей?

**Ответ:** Зависит от системы. Типичные кандидаты: база данных (соединения, запросы), API (CPU, память), статика (пропускная способность). Сначала профилировать и измерять. Для frontend — CDN для статики, пагинация/виртуализация для больших списков. Для backend — индексы БД, кэш (Redis), горизонтальное масштабирование за load balancer. Stateless backend позволяет добавлять инстансы без переделки.

### Как масштабировать read vs write?

**Ответ:** Read обычно преобладает. Репликация — несколько read-replica для чтения, один primary для записи. Кэш (Redis) перед БД для частых read. CQRS — разделение моделей чтения и записи. Write — шардинг по ключу (user_id, tenant_id). Очереди для тяжёлых записей (асинхронная обработка).

### Когда переходить от монолита к микросервисам?

**Ответ:** Не по умолчанию. Микросервисы — когда монолит реально мешает: разные команды блокируют друг друга, части системы масштабируются по-разному, нужны разные технологии. Сначала — модульный монолит (чёткие границы внутри одного приложения). Переход когда: команды >5–10, деплой одной части блокирует остальные, нужна независимая масштабируемость компонентов.
