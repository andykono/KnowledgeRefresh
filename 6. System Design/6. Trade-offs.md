## Зачем на собеседовании

Нет идеальных решений — есть компромиссы. Важно уметь обосновывать выбор и понимать последствия.

## Client-side vs Server-side rendering

| | CSR | SSR | SSG |
| :-- | :-- | :-- | :-- |
| **FCP** | Медленнее (ждём JS) | Быстрее | Самый быстрый |
| **TTI** | Позже | Раньше | Раньше |
| **SEO** | Сложнее | Хорошо | Хорошо |
| **Сложность** | Ниже | Выше | Средняя |
| **Кэширование** | Статика | Сложнее | Отлично |
| **Персонализация** | Легко | На каждый запрос | Сложнее |

**Когда CSR:** SPA, личный кабинет, много интерактива.  
**Когда SSR:** SEO, быстрый FCP, контентные страницы.  
**Когда SSG:** блоги, документация, статичный контент.

## Data structures

| Структура | Плюсы | Минусы |
| :-- | :-- | :-- |
| **Array** | Доступ O(1), порядок | Вставка в начало O(n), поиск O(n) |
| **Map/Set** | Поиск O(1), уникальность | Больше памяти |
| **Linked List** | Вставка в начало O(1) | Доступ O(n) |
| **Tree** | Иерархия, поиск O(log n) в BST | Сложнее реализация |

Выбор зависит от операций: частый доступ по индексу → array; частый поиск → Map; иерархия → tree.

## Caching strategies

| Стратегия | Когда использовать |
| :-- | :-- |
| **Cache-first** | Статика, редко меняется |
| **Network-first** | Актуальность важнее скорости |
| **Stale-while-revalidate** | Баланс: быстрый ответ + обновление |
| **Cache-only** | Offline, критичные ресурсы |

**Trade-off:** свежесть vs скорость. Больше кэша — быстрее, но риск устаревших данных.

## Consistency vs Availability (CAP)

В распределённых системах при сетевом сбое нельзя одновременно гарантировать:
- **Consistency** — все узлы видят одни данные
- **Availability** — каждый запрос получает ответ

Выбор зависит от домена: банк — consistency, соцсеть — availability.

## Модульность vs Простота

- Больше модулей — гибкость, но сложнее навигация и зависимости
- Меньше модулей — проще, но выше связанность
- Имеет смысл вводить модули при реальном повторении и росте кода

## Вопросы для обсуждения

### Почему выбран именно этот подход?

**Ответ:** Структурированно: контекст (ограничения, требования) → варианты → критерии выбора → решение. Пример: «Нужен быстрый FCP и SEO → рассматривали CSR, SSR, SSG → приоритет скорость и статичный контент → выбрали SSG». Или: «Много read, мало write → репликация + кэш для read, один primary для write». Показать, что выбор обоснован контекстом, а не привычкой.

### Какие альтернативы рассматривались?

**Ответ:** Назвать 2–3 варианта и почему отброшены. Пример: «SSR vs CSR — SSR выбран из-за SEO и FCP; CSR отброшен, т.к. контент должен индексироваться». Или: «Redux vs Zustand — Zustand выбран за простоту и размер; Redux — для сложного state с middleware, у нас проще». Показать осознанный выбор, а не случайность.

### При каких условиях решение перестанет подходить?

**Ответ:** Честно обозначить границы. Примеры: «SSG — перестанет подходить при частой персонализации или real-time данных». «Монолит — при росте до 5+ команд, когда деплой блокирует всех». «Кэш-first — при требовании всегда свежих данных». «OnPush (Angular) — при сложных цепочках обновлений». Показать, что вы понимаете ограничения и готовы пересмотреть решение при изменении контекста.
