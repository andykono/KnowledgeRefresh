# 5.6 Testing Best Practices

[[5. Testing.md]]

Guidelines for writing maintainable, valuable tests that catch real bugs without slowing development.

---

## What to Test

| Do Test | Why |
| :-- | :-- |
| **User-visible behavior** | Tests reflect real usage |
| **Critical paths** | Login, checkout, data integrity |
| **Edge cases** | Empty states, errors, boundaries |
| **Accessibility** | Labels, roles, keyboard nav |
| **Pure logic** | Reducers, validators, formatters |
| **Custom hooks** | Reusable state logic |

---

## What NOT to Test

| Don't Test | Why |
| :-- | :-- |
| **Implementation details** | Internal state, private methods |
| **Third-party libraries** | React, lodash, etc. |
| **Trivial code** | Simple getters, constants |
| **Every branch** | Focus on important paths |
| **Styling** | Use visual regression sparingly |

### Anti-Pattern: Testing Implementation

```typescript
// ❌ Bad: Testing internal state
test('count state is 0', () => {
  const { result } = renderHook(() => useCounter());
  expect(result.current._internalCount).toBe(0); // Private!
});

// ✅ Good: Test public behavior
test('displays 0 initially', () => {
  const { result } = renderHook(() => useCounter());
  expect(result.current.count).toBe(0);
});
```

### Anti-Pattern: Testing Implementation Details

```typescript
// ❌ Bad: Testing that useState was called
test('uses useState', () => {
  const useStateSpy = vi.spyOn(React, 'useState');
  render(<Counter />);
  expect(useStateSpy).toHaveBeenCalled();
});

// ✅ Good: Test rendered output
test('shows count', () => {
  render(<Counter />);
  expect(screen.getByText('0')).toBeInTheDocument();
});
```

## Testing Pyramid Ratios

Aim for this distribution:

| Level | Ratio | Count (example) |
| :-- | :-- | :-- |
| Unit | 60–80% | 100+ tests |
| Integration | 20–30% | 30–50 tests |
| E2E | 5–10% | 5–15 tests |

More unit tests = faster feedback. Fewer E2E = less flakiness.

---

## Coverage Goals

| Metric | Target | Notes |
| :-- | :-- | :-- |
| Statements | 80%+ | Baseline |
| Branches | 70%+ | Harder to reach |
| Functions | 80%+ | Ensure functions are exercised |
| Lines | 80%+ | Similar to statements |

```typescript
// vitest.config.ts
export default defineConfig({
  test: {
    coverage: {
      provider: 'v8',
      reporter: ['text', 'html'],
      exclude: ['**/*.test.ts', '**/test/**'],
      thresholds: {
        statements: 80,
        branches: 70,
        functions: 80,
        lines: 80,
      },
    },
  },
});
```

**Warning:** High coverage ≠ good tests. 100% coverage with weak assertions can miss bugs. Prefer meaningful tests over coverage targets.

## Testing Anti-Patterns

### 1. Brittle Selectors

```typescript
// ❌ Bad: Breaks when text changes
screen.getByText('Submit');

// ✅ Better: Role + name
screen.getByRole('button', { name: /submit/i });
```

### 2. Testing Multiple Things

```typescript
// ❌ Bad: One test, many assertions
test('form works', () => {
  render(<Form />);
  expect(screen.getByLabelText('Email')).toBeInTheDocument();
  expect(screen.getByLabelText('Password')).toBeInTheDocument();
  // ... 10 more assertions
});

// ✅ Good: One behavior per test
test('shows email field', () => {
  render(<Form />);
  expect(screen.getByLabelText('Email')).toBeInTheDocument();
});
test('validates empty email', async () => { /* ... */ });
```

### 3. Sleeping Instead of Waiting

```typescript
// ❌ Bad
await new Promise((r) => setTimeout(r, 1000));

// ✅ Good
await waitFor(() => expect(screen.getByText('Loaded')).toBeInTheDocument());
// or
await screen.findByText('Loaded');
```

### 4. Over-Mocking

```typescript
// ❌ Bad: Mocking everything
vi.mock('./Button');
vi.mock('./Input');
vi.mock('./Form');
// Test passes but catches nothing

// ✅ Good: Mock only external deps (API, router)
```

## Accessibility Testing with jest-axe

```bash
npm install -D jest-axe @types/jest-axe
```

```typescript
// src/test/setup.ts
import 'jest-axe/extend-expect';
```

```typescript
import { render, screen } from '@testing-library/react';
import { axe, toHaveNoViolations } from 'jest-axe';
import { Button } from './Button';

expect.extend(toHaveNoViolations);

describe('Button', () => {
  it('has no accessibility violations', async () => {
    const { container } = render(<Button>Click me</Button>);
    const results = await axe(container);
    expect(results).toHaveNoViolations();
  });

  it('is accessible with aria-label', async () => {
    const { container } = render(
      <button aria-label="Close">
        <span aria-hidden>×</span>
      </button>
    );
    const results = await axe(container);
    expect(results).toHaveNoViolations();
  });
});
```

## Snapshot Testing: When Useful vs Harmful

### ⚠️ Use Sparingly

**Useful for:**
- Error messages or complex error UIs
- Serialization formats (JSON output)
- Low-churn, stable components

**Harmful when:**
- Large DOM snapshots (brittle, noisy)
- Components change often
- Snapshots updated blindly without review

```typescript
// ✅ Reasonable: Small, stable output
test('error message format', () => {
  const error = formatError({ code: 'E001', message: 'Not found' });
  expect(error).toMatchSnapshot();
});

// ❌ Avoid: Full component snapshot
test('renders', () => {
  const { container } = render(<ComplexForm />);
  expect(container).toMatchSnapshot(); // 500 lines, breaks on any change
});
```

**Best practice:** Prefer explicit assertions over snapshots. Use snapshots only for stable, small outputs.

## CI Pipeline Setup

```yaml
# .github/workflows/test.yml
name: Test

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  unit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm run test -- --coverage
      - run: npm run test:e2e  # if Playwright configured

  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm run lint
      - run: npm run typecheck  # tsc --noEmit
```

```json
// package.json
{
  "scripts": {
    "test": "vitest run",
    "test:watch": "vitest",
    "test:coverage": "vitest run --coverage",
    "test:e2e": "playwright test",
    "lint": "eslint src --ext .ts,.tsx",
    "typecheck": "tsc --noEmit"
  }
}
```

## Test Naming Conventions

```typescript
// Pattern: it('does X when Y')
it('shows error when email is invalid', () => {});
it('redirects to dashboard when login succeeds', () => {});
it('calls onSubmit with form data when submit is clicked', () => {});

// Or: it('should X')
it('should display loading state while fetching', () => {});
```

Be descriptive. A failing test name should tell you what broke.
