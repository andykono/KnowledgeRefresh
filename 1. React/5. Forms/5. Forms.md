## Form Architecture at a Glance

### Controlled (React owns state)
```typescript
const [value, setValue] = useState('');
<input value={value} onChange={e => setValue(e.target.value)} />
```

### Uncontrolled (DOM owns state)
```typescript
const ref = useRef<HTMLInputElement>(null);
<input ref={ref} defaultValue="initial" />
// Access: ref.current?.value
```

### React Hook Form (hybrid, performant)
```typescript
const { register, handleSubmit } = useForm();
<input {...register('email')} />
onSubmit={handleSubmit(data => console.log(data))}
```

## Key Concepts

### State Management
- **Controlled**: Every keystroke triggers re-render; full control for validation/transform
- **Uncontrolled**: DOM holds value; fewer re-renders; good for simple cases
- **React Hook Form**: Uncontrolled under the hood with controlled API; minimal re-renders

### Validation Strategies
- **Client-side**: Zod, Yup, custom validators—validate before submit
- **Real-time**: Validate on blur, change, or debounced
- **Server-side**: Always validate on backend; client validation is UX only

### TypeScript Integration
- Form event types: `ChangeEvent<HTMLInputElement>`, `FormEvent`, etc.
- Generic form types: `UseFormRegisterReturn`, `FieldValues`
- Schema inference: Zod/Yup → TypeScript types

## Best Practices

✅ **DO:**
- Use controlled components when you need validation or transformation
- Use React Hook Form for complex forms (better performance)
- Validate on both client and server
- Use schema validation (Zod/Yup) for type safety
- Provide clear error messages and accessibility (aria-invalid, aria-describedby)

❌ **DON'T:**
- Don't use controlled for every input if uncontrolled suffices
- Don't skip server-side validation
- Don't validate only on submit—consider blur/change for UX
- Don't forget `htmlFor`/`id` for labels
