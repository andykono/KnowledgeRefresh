# Rendering & Reconciliation

[[1. React Core Concepts.md]]

---

Understanding how React renders and updates the DOM is critical for performance.

## React Rendering Process

### 1. Render Phase
- React calls component functions
- Returns JSX describing the UI
- Creates a virtual representation (virtual DOM)

### 2. Reconciliation Phase
- React compares old virtual DOM with new virtual DOM
- Determines what changed (diffing algorithm)
- Generates list of DOM updates

### 3. Commit Phase
- React updates the actual DOM
- Runs useEffect cleanup and setup
- Updates refs

```typescript
function Component() {
  console.log('1. Render phase - component function called');

  useEffect(() => {
    console.log('3. Commit phase - effect runs after DOM updated');
    return () => {
      console.log('3b. Cleanup - before next render');
    };
  });

  return <div>Content</div>;  // 2. Reconciliation with virtual DOM
}

// Output:
// 1. Render phase - component function called
// 3. Commit phase - effect runs after DOM updated
// (on re-render)
// 1. Render phase - component function called
// 3b. Cleanup - before next render
// 3. Commit phase - effect runs after DOM updated
```

## Virtual DOM & Diffing

React uses a **virtual DOM** - a JavaScript representation of the actual DOM.

```typescript
// Virtual DOM (what React creates)
const vdom = {
  type: 'div',
  props: { className: 'container' },
  children: [
    { type: 'h1', props: {}, children: ['Hello'] },
    { type: 'p', props: {}, children: ['Content'] }
  ]
};

// React compares old vdom with new vdom
// Only updates changed parts in real DOM
```

### Diffing Algorithm

```typescript
// Before
<div>
  <p key="a">Item A</p>
  <p key="b">Item B</p>
</div>

// After (moved order)
<div>
  <p key="b">Item B</p>
  <p key="a">Item A</p>
</div>

// Without keys: React re-creates both elements
// With keys: React moves them (more efficient)
```

## Keys & List Rendering

**Keys help React identify which items changed.**

```typescript
// ❌ BAD: Using index as key
{items.map((item, index) => (
  <div key={index}>{item.name}</div>
))}

// ❌ Problem: If list reorders, keys stay the same
// Item 1 (key=0) → Item 2 (key=0)
// React thinks it's the same component, but props changed

// ✅ GOOD: Using unique ID
{items.map(item => (
  <div key={item.id}>{item.name}</div>
))}

// ✅ React knows each item uniquely
// If reordered, React moves the elements correctly
```

## Conditional Rendering

Different ways to render conditionally:

```typescript
// If statement (outside JSX)
function Component({ isLoggedIn }: { isLoggedIn: boolean }) {
  if (isLoggedIn) {
    return <Dashboard />;
  }
  return <LoginPage />;
}

// Ternary operator
<div>{isLoggedIn ? <Dashboard /> : <LoginPage />}</div>

// Logical AND (render if true)
<div>{isLoggedIn && <Dashboard />}</div>

// Switch statement
function Component({ page }: { page: 'home' | 'about' | 'contact' }) {
  switch (page) {
    case 'home':
      return <HomePage />;
    case 'about':
      return <AboutPage />;
    case 'contact':
      return <ContactPage />;
  }
}

// Element variable (complex logic)
let content;
if (isLoading) {
  content = <Spinner />;
} else if (error) {
  content = <Error />;
} else {
  content = <Data />;
}
return <div>{content}</div>;
```

## Re-render Triggers

A component re-renders when:

1. **Its state changes**
   ```typescript
   const [count, setCount] = useState(0);
   setCount(1);  // Triggers re-render
   ```

2. **Its props change**
   ```typescript
   // Parent re-renders, child gets new props
   <Child value={newValue} />
   ```

3. **Parent re-renders**
   ```typescript
   // If parent re-renders, all children re-render
   function Parent() {
     const [state, setState] = useState();
     return <Child />;  // Re-renders when parent re-renders
   }
   ```

4. **useReducer state changes**
   ```typescript
   const [state, dispatch] = useReducer(reducer, initial);
   dispatch(action);  // Triggers re-render
   ```

## Render vs Commit

### Safe in Render Phase:
- ✅ Component functions
- ✅ Returning JSX
- ✅ Calculating values

### Unsafe in Render Phase:
- ❌ Side effects (API calls, subscriptions)
- ❌ Updating state/props
- ❌ DOM mutations
- ❌ setTimeout/setInterval

### Safe in Commit Phase (useEffect):
- ✅ Side effects
- ✅ DOM mutations
- ✅ API calls
- ✅ Subscriptions
- ✅ setTimeout/setInterval

## Performance: Preventing Unnecessary Renders

```typescript
// ❌ BAD: Causes re-render even if props didn't change
function Component({ data }: { data: any }) {
  return <Child item={data} />;
}

// ✅ GOOD: Memoized to prevent unnecessary renders
import { memo } from 'react';

const Component = memo(function Component({ data }: { data: any }) {
  return <Child item={data} />;
});

// ✅ GOOD: Even better with useCallback for callbacks
function Parent() {
  const [count, setCount] = useState(0);

  const handleClick = useCallback(() => {
    setCount(c => c + 1);
  }, []);  // Dependency array

  return <Child onClick={handleClick} />;
}
```

## Rendering Best Practices

✅ **DO:**
- Use unique keys in lists
- Return consistent element types
- Move state down to where it's needed
- Split large components into smaller ones

❌ **DON'T:**
- Don't overuse memo or useMemo
- Don't create objects/functions inside render
- Don't have side effects in component body
- Don't mutate state directly
