# Keys and Refs

[[1. React Core Concepts.md]]

---

Keys identify list items. Refs access DOM directly.

## Keys in Lists

```typescript
// ✅ CORRECT: Unique ID as key
{items.map(item => (
  <div key={item.id}>{item.name}</div>
))}

// ❌ WRONG: Index as key (bad performance)
{items.map((item, index) => (
  <div key={index}>{item.name}</div>
))}
```

**Why:** React uses keys to match items. Without unique keys, reordering breaks component state.

---

## Why Unique Keys Matter

React uses keys to determine which items have changed, been added, or removed during reconciliation. When keys are stable and unique:

- **Efficient updates:** React can match old and new items by key, minimizing DOM operations
- **State preservation:** Component state stays tied to the correct item when the list reorders
- **Animation correctness:** CSS transitions and animations target the right elements

Without unique keys, React falls back to index-based matching. Inserting an item at the start shifts all indices, causing React to treat every item as "changed" and potentially re-mount components, losing local state.

```typescript
// Problem: Adding item at start with index keys
// Before: [A(0), B(1), C(2)] → After: [X(0), A(1), B(2), C(3)]
// React thinks A→X, B→A, C→B, and creates new C — wrong!
```

---

## Key Prop Internals

Keys are used internally by React's reconciliation algorithm:

- Keys are **not passed** as props to the component — they're used by React only
- Keys must be **unique among siblings**, not globally
- Keys should be **stable** across renders — avoid `Math.random()` or `Date.now()`
- Changing a key forces React to **unmount** the old component and **mount** a new one

```typescript
// Keys are sibling-scoped
function Parent() {
  return (
    <>
      <Child key="a" />  {/* key in this list */}
      <Child key="b" />
    </>
  );
}

// Different parent = different key scope
function AnotherParent() {
  return <Child key="a" />;  // Same key, different scope — OK
}
```

---

## Resetting Component State with Keys

Because changing a key forces a full remount, you can use keys to **reset** component state:

```typescript
function UserProfile({ userId }: { userId: string }) {
  const [editing, setEditing] = useState(false);
  // When userId changes, we want fresh state
  return <ProfileForm key={userId} />;
}

// Reset form when switching tabs
function TabbedForm() {
  const [activeTab, setActiveTab] = useState('personal');
  return (
    <Form key={activeTab} />  // New key = fresh mount = reset state
  );
}
```

This pattern is useful when switching between distinct entities (e.g., different user IDs, different form modes) where you want to discard previous state.

---

## When Index Keys Are OK

Index as key is acceptable when:

- The list is **static** (no reordering, filtering, or insertion)
- List items have **no meaningful identity** (e.g., pure display items)
- Items have **no internal state** or the state is trivial

```typescript
// OK: Static list, no reordering, no filtering
const staticLabels = ['Name', 'Email', 'Phone'];
{staticLabels.map((label, i) => <div key={i}>{label}</div>)}

// OK: Items are purely presentational, no state
{items.map((item, i) => <span key={i}>{item}</span>)}
```

When in doubt, prefer a unique ID. If you don't have one, generate a stable key from the item's content (e.g., `key={item.id ?? item.name}`) or use a library like `uuid` for new items.

## Refs

```typescript
const inputRef = useRef<HTMLInputElement>(null);

const focusInput = () => {
  inputRef.current?.focus();
};

return (
  <>
    <input ref={inputRef} />
    <button onClick={focusInput}>Focus</button>
  </>
);
```

Use refs sparingly - only for:
- Focus/blur
- Measuring DOM
- Triggering animations
- Integrating with third-party code

---

## useRef for DOM Access

```typescript
const inputRef = useRef<HTMLInputElement>(null);
const videoRef = useRef<HTMLVideoElement>(null);

useEffect(() => {
  inputRef.current?.focus();
}, []);

const playVideo = () => videoRef.current?.play();
```

Refs are populated after the component mounts. For function components, `ref.current` is `null` during the initial render. Use `useEffect` or event handlers to access DOM after mount.

---

## useRef for Mutable Values

Refs can hold any mutable value that persists across renders without triggering re-renders:

```typescript
const renderCount = useRef(0);
const previousValue = useRef<string | undefined>();

useEffect(() => {
  renderCount.current += 1;
});

useEffect(() => {
  previousValue.current = value;
}, [value]);
```

Unlike state, updating `ref.current` does **not** cause a re-render. Use this for timers, subscriptions, previous values, or any value you need to persist but not display.

---

## forwardRef Pattern

To pass a ref to a custom component, the component must forward it with `forwardRef`:

```typescript
const FancyInput = forwardRef<HTMLInputElement, { label: string }>(
  (props, ref) => {
    return (
      <div>
        <label>{props.label}</label>
        <input ref={ref} {...props} />
      </div>
    );
  }
);

// Parent can now get ref to the inner input
const parentRef = useRef<HTMLInputElement>(null);
<FancyInput ref={parentRef} label="Name" />
```

---

## useImperativeHandle

Expose a custom API to parent refs instead of the raw DOM node:

```typescript
const FancyInput = forwardRef<HTMLInputElement, { label: string }>(
  (props, ref) => {
    const inputRef = useRef<HTMLInputElement>(null);

    useImperativeHandle(ref, () => ({
      focus: () => inputRef.current?.focus(),
      scrollIntoView: () => inputRef.current?.scrollIntoView(),
      getValue: () => inputRef.current?.value ?? '',
    }));

    return <input ref={inputRef} {...props} />;
  }
);

// Parent uses custom API
const ref = useRef<{ focus: () => void; getValue: () => string }>(null);
ref.current?.focus();   // Custom method, not DOM
ref.current?.getValue();
```

---

## Callback Refs

A function ref receives the DOM node when it mounts and `null` when it unmounts:

```typescript
const [height, setHeight] = useState(0);

const measuredRef = useCallback((node: HTMLDivElement | null) => {
  if (node !== null) {
    setHeight(node.getBoundingClientRect().height);
  }
}, []);

return <div ref={measuredRef}>Measure me</div>;
```

Useful for measuring elements or integrating with libraries that need the DOM node. React calls the callback with the node on mount and with `null` on unmount.

---

## Ref Cleanup (React 19)

In React 19, ref callbacks receive `null` when the element is removed, enabling cleanup:

```typescript
const ref = useCallback((node: HTMLDivElement | null) => {
  if (node) {
    const observer = new ResizeObserver((entries) => {
      // handle resize
    });
    observer.observe(node);
    // React 19: cleanup when node becomes null
    return () => observer.disconnect();
  }
}, []);
```

This pattern ensures proper cleanup of observers, event listeners, or other side effects tied to the DOM node.
