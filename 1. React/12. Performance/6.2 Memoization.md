# Memoization

[[6. Performance.md]]

---

Memoization prevents unnecessary work by caching results. In React, we memoize components, values, and callbacks.

## React.memo Deep Dive

`React.memo` wraps a component and skips re-rendering when props are shallowly equal.

```typescript
import { memo } from 'react';

interface ProductCardProps {
  id: string;
  name: string;
  price: number;
}

const ProductCard = memo(function ProductCard({ id, name, price }: ProductCardProps) {
  return (
    <div className="product-card">
      <h3>{name}</h3>
      <p>${price}</p>
    </div>
  );
});
```

### Custom Comparator

When props include objects/arrays, shallow comparison fails. Use a custom comparator:

```typescript
const ProductCard = memo(
  function ProductCard({ product }: { product: Product }) {
    return <div>{product.name}</div>;
  },
  (prevProps, nextProps) => {
    // Return true if props are equal (skip re-render)
    return prevProps.product.id === nextProps.product.id &&
           prevProps.product.updatedAt === nextProps.product.updatedAt;
  }
);
```

### When to Use memo

- **Use:** Expensive leaf components, list items, components that re-render often with same props
- **Avoid:** Cheap components, components that always get new props

```typescript
// Good candidate: expensive to render, receives stable props
const Chart = memo(function Chart({ data }: { data: DataPoint[] }) {
  return <ComplexSVG data={data} />;
});

// Bad candidate: parent passes new object every time
<Chart data={[...items]} />  // New array reference = memo useless
```

## useMemo

Memoizes the **result** of an expensive computation.

### When to Use useMemo

- Expensive calculations (filtering, sorting, transformations)
- Referential equality for objects/arrays passed to memoized children
- When the computation depends on specific inputs

```typescript
function ProductList({ products, filter }: Props) {
  const filteredProducts = useMemo(() => {
    return products.filter(p =>
      p.name.toLowerCase().includes(filter.toLowerCase())
    );
  }, [products, filter]);

  return (
    <ul>
      {filteredProducts.map(p => (
        <ProductCard key={p.id} product={p} />
      ))}
    </ul>
  );
}
```

### When NOT to Use useMemo

- Trivial computations (no measurable cost)
- When dependencies change every render anyway
- Premature optimization

```typescript
// Overkill
const doubled = useMemo(() => count * 2, [count]);

// Better
const doubled = count * 2;
```

### Referential Stability

```typescript
// Unstable: new object every render
const config = { threshold: 0.5 };
<ExpensiveChild config={config} />  // Re-renders every time

// Stable with useMemo
const config = useMemo(() => ({ threshold: 0.5 }), []);
<ExpensiveChild config={config} />  // Re-renders only when config deps change
```

## useCallback

Memoizes a **function** so it keeps the same reference across renders.

### Event Handlers

```typescript
function Parent() {
  const [count, setCount] = useState(0);

  const handleClick = useCallback(() => {
    setCount(c => c + 1);
  }, []);  // Stable reference

  return <ExpensiveChild onClick={handleClick} />;
}
```

### The Ref Pattern

Sometimes you need a callback that always has latest values without causing re-renders. Use a ref:

```typescript
function useStableCallback<T extends (...args: unknown[]) => unknown>(fn: T): T {
  const ref = useRef(fn);
  ref.current = fn;  // Always up to date

  return useCallback((...args: Parameters<T>) => {
    return ref.current(...args);
  }, []) as T;
}

// Usage: callback never changes reference, but always has latest closure
function SearchInput() {
  const [query, setQuery] = useState('');
  const onSearch = useStableCallback(() => {
    fetchResults(query);  // Always gets latest query
  });
  return <SearchBar onSearch={onSearch} />;
}
```

### When useCallback Helps

- Passing callbacks to memoized children
- Callbacks in dependency arrays (useEffect, useMemo)
- Callbacks passed to native elements (rarely needed)

### When useCallback Doesn't Help

- Callbacks only used in current component (no child re-render concern)
- Child isn't memoized

## React Compiler (Auto-Memoization)

The React Compiler (experimental) automatically memoizes components and hooks when safe.

```typescript
// Without compiler: might need memo
function ProductCard({ product }: { product: Product }) {
  return <div>{product.name}</div>;
}

// With React Compiler: automatically optimized
// No manual memo/useMemo/useCallback needed in many cases
```

### Enabling React Compiler

```json
// package.json
{
  "devDependencies": {
    "babel-plugin-react-compiler": "latest"
  }
}
```

```javascript
// babel.config.js
module.exports = {
  plugins: [
    ['react-compiler', {
      target: '18', // or '19'
    }],
  ],
};
```

The compiler analyzes your code and adds memoization where it's safe. It cannot memoize across module boundaries or when refs are used in complex ways.

## Memoization Decision Flow

```
Is the component/calculation expensive?
├─ No → Don't memoize
└─ Yes
   ├─ Component with stable props? → React.memo
   ├─ Expensive computation? → useMemo
   └─ Callback passed to memoized child? → useCallback
```

## Common Pitfalls

```typescript
// Pitfall 1: Inline objects defeat memo
<Child style={{ color: 'red' }} />  // New object every render

// Pitfall 2: useCallback with changing deps
const fn = useCallback(() => doSomething(id), [id]);
// fn changes when id changes — only use if child needs latest id

// Pitfall 3: Memoizing everything
// Adds overhead; use only when profiling shows benefit
```
