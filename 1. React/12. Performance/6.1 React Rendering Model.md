# React Rendering Model

[[6. Performance.md]]

---

Understanding how React renders is foundational to performance optimization. React's rendering follows a predictable three-phase cycle.

## The Three Phases: Trigger → Render → Commit

### 1. Trigger

Something causes React to schedule a render:

```typescript
// State update (most common)
const [count, setCount] = useState(0);
setCount(1); // Triggers render

// Parent re-render
function Parent() {
  const [state, setState] = useState(0);
  return <Child />; // Child re-renders when Parent does
}

// Context value change
const ThemeContext = createContext('light');
// Any consumer re-renders when provider value changes
```

### 2. Render

React calls your component functions to compute the new UI. This phase is **pure** — no side effects, no DOM updates.

```typescript
function ProductList({ products }: { products: Product[] }) {
  // This runs during Render phase
  const filtered = products.filter(p => p.inStock);
  return (
    <ul>
      {filtered.map(p => (
        <li key={p.id}>{p.name}</li>
      ))}
    </ul>
  );
}
```

### 3. Commit

React applies the computed changes to the DOM. This is when `useLayoutEffect` runs (synchronously) and `useEffect` is scheduled (asynchronously).

```typescript
function Modal({ isOpen }: { isOpen: boolean }) {
  useEffect(() => {
    // Runs after Commit — DOM is updated
    if (isOpen) {
      document.body.style.overflow = 'hidden';
    }
    return () => {
      document.body.style.overflow = '';
    };
  }, [isOpen]);
  return isOpen ? <div className="modal">...</div> : null;
}
```

## Reconciliation

React uses a **diffing algorithm** to determine what changed between renders. It compares the previous tree with the new one.

### Key Reconciliation Rules

1. **Same position, same type** → Update in place
2. **Same position, different type** → Tear down old, mount new
3. **Different position** → Tear down old, mount new (even if same component)

```typescript
// BAD: Switching component type at same position
{isLoggedIn ? <UserProfile /> : <LoginForm />}
// React unmounts one, mounts the other — loses internal state

// BETTER: Preserve identity with key
{items.map(item => (
  <Item key={item.id} data={item} />
))}
// key helps React match items across renders
```

### The Virtual DOM

React maintains an in-memory representation (Virtual DOM / Fiber tree) of your UI. It diffs this tree to compute minimal DOM updates.

```typescript
// React doesn't re-render the whole tree
// It computes: "Only this <span> text changed"
function Counter() {
  const [count, setCount] = useState(0);
  return (
    <div>
      <h1>Count: {count}</h1>  {/* Only this updates */}
      <button onClick={() => setCount(c => c + 1)}>+</button>
    </div>
  );
}
```

## Fiber Architecture Basics

React 16+ uses **Fiber** — a unit of work. Each component instance has a Fiber node.

- **Fiber** = lightweight JavaScript object representing a unit of work
- **Scheduling** = React can pause, resume, prioritize work
- **Concurrent features** = useTransition, useDeferredValue rely on this

```typescript
// Fiber enables concurrent rendering
function SearchResults({ query }: { query: string }) {
  const deferredQuery = useDeferredValue(query);
  // React can show stale results while new ones load
  const results = useMemo(
    () => search(deferredQuery),
    [deferredQuery]
  );
  return <List items={results} />;
}
```

## What Causes Re-renders

| Cause | Example | Mitigation |
| :-- | :-- | :-- |
| State change | `setState` | Expected — optimize children |
| Parent re-render | Parent's state/context changed | `React.memo` on children |
| Context change | Provider value changed | Split contexts, memo values |
| Props change | Parent passes new object/array | Memoize in parent |
| Hooks dependency | useEffect/useMemo deps | Stabilize dependencies |

```typescript
// Context causes all consumers to re-render
const AppContext = createContext({ user: null, theme: 'light' });

function App() {
  const [user, setUser] = useState(null);
  const [theme, setTheme] = useState('light');
  const value = { user, theme }; // New object every render!
  return (
    <AppContext.Provider value={value}>
      <Header />  {/* Re-renders when user OR theme changes */}
    </AppContext.Provider>
  );
}

// Fix: Split contexts or memoize value
const UserContext = createContext(null);
const ThemeContext = createContext('light');
```

## React.StrictMode Behavior

In development, StrictMode **double-invokes** component functions, effects, and some lifecycle methods to surface impure code.

```typescript
// StrictMode runs this twice in dev
function Component() {
  console.log('render'); // Logs twice in dev
  useEffect(() => {
    console.log('effect'); // Logs twice in dev
    return () => console.log('cleanup'); // Cleanup runs between
  }, []);
  return <div>Hello</div>;
}
```

**Why?** To help you find:
- Side effects in render
- Missing effect cleanup
- Deprecated APIs

**Production:** StrictMode does not double-invoke. Your app runs once.

## Batching Updates

React batches state updates for performance.

```typescript
function Counter() {
  const [a, setA] = useState(0);
  const [b, setB] = useState(0);

  const handleClick = () => {
    setA(a + 1);
    setB(b + 1);
    // Only one re-render! (React 18+ batches automatically)
  };
}
```

In React 18+, batching works in event handlers, setTimeout, fetch callbacks, etc.
