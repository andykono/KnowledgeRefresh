React Server Components (RSC) run on the server by default. They enable zero-bundle components, direct database/API access, and streaming. This section covers the RSC architecture, server vs client boundaries, Server Actions, and streaming with Suspense.

## Quick Concepts

### Server Components (Default)
- Run only on the server
- No `useState`, `useEffect`, or browser APIs
- Can fetch data directly, access DB, use secrets
- Zero JavaScript sent to client for these components

### Client Components (`"use client"`)
- Run on server for SSR, then hydrate on client
- Can use hooks, event handlers, browser APIs
- JavaScript is bundled and sent to client

### Server Actions (`"use server"`)
- Async functions that run on the server
- Can be called from Client Components
- Used for mutations, form submissions, revalidation

---

## When to Use What

```
Need hooks, onClick, useState?           → "use client"
Need to fetch data in component?         → Server Component (default)
Need to mutate data (form submit, etc)?  → Server Action
Need streaming / progressive loading?    → Suspense + RSC
```

---

## Data Flow

```
Server Component
  → fetches data (async)
  → passes to Client Component as props
  → Client Component uses hooks, events
  → Client calls Server Action for mutations
  → Server Action runs on server, can revalidate
```

---

## Serialization Rules

Server Components can only pass **serializable** props to Client Components:
- ✅ Primitives, plain objects, arrays
- ✅ JSX (passed as children)
- ❌ Functions (except Server Actions)
- ❌ Class instances, Symbols, etc.
