## Common Implementation Decisions

### State Management
```typescript
// Simple: useState + useContext
✅ Use for: Small to medium apps, learning

// Medium: Custom hooks + Context
✅ Use for: Growing apps, specific features

// Complex: Redux / Zustand
✅ Use for: Large apps, complex flows
```

### Data Fetching
```typescript
// Simple: useEffect + useState
✅ Use for: Learning, simple cases

// Better: Custom hook (useAsync)
✅ Use for: Most cases, reusable

// Best: TanStack Query
✅ Use for: Complex caching, background sync
```

### Form Handling
```typescript
// Simple: useState for each field
✅ Use for: Very simple forms

// Better: Custom hook (useForm)
✅ Use for: Standard forms

// Best: react-hook-form + validation
✅ Use for: Complex forms, performance critical
```

## Scaling Patterns

As your project grows:

```
Small App (MVP)
  ↓
Use: Component-based + basic hooks
    ↓
Growing App (10+ pages)
  ↓
Use: FSD + custom hooks + Context
    ↓
Large App (50+ pages)
  ↓
Use: FSD + Redux/Zustand + TanStack Query
    ↓
Enterprise App
  ↓
Use: Clean Architecture + advanced patterns
```

## Best Practices Learned from Examples

### Folder Structure
- ✅ Organize by features, not file types
- ✅ Keep API calls in separate modules
- ✅ Use index.ts for clean exports
- ✅ Group related types in model/

### State Management
- ✅ Keep state as close as possible
- ✅ Use Context for app-wide values
- ✅ Derive state instead of storing
- ✅ Normalize complex data

### Component Design
- ✅ One responsibility per component
- ✅ Props should be data or callbacks
- ✅ Use composition over inheritance
- ✅ Lift state when needed

### Testing
- ✅ Test behavior, not implementation
- ✅ Mock external dependencies
- ✅ Write integration tests for flows
- ✅ Don't test library code

## Common Pitfalls to Avoid

❌ **Props Drilling**
```typescript
// Bad: Pass through multiple levels
<Component user={user} setUser={setUser}>
  <Child user={user} setUser={setUser}>
    <GrandChild user={user} setUser={setUser} />
  </Child>
</Component>

// Good: Use Context
```

❌ **Global State for Everything**
```typescript
// Bad: All state in Redux
const state = { /* 500 things */ };

// Good: Keep local state local
// Only global state: user, theme, language
```

❌ **Over-Engineering**
```typescript
// Bad: Complex patterns for simple needs
// Hook + reducer + Context + saga for toggle

// Good: Simple solution for simple problem
const [isOpen, setIsOpen] = useState(false);
```
