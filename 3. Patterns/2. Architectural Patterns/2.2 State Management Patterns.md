# State Management Patterns

Паттерны для управления сложным состоянием приложения. Обзор популярных решений.

## When Do You Need State Management?

```
Simple App
├── useState + useContext
└── Good enough!

Growing App
├── Custom hooks
├── Multiple features
└── Custom hooks + Context

Complex App
├── Redux / Zustand / Jotai
├── Complex flows
├── Multiple developers
└── Need debugging tools
```

## Redux Pattern

**Classic** predictable state management with actions and reducers.

### Key Concepts
- **Store:** Single source of truth
- **Actions:** Objects describing what happened
- **Reducers:** Pure functions updating state
- **Selectors:** Accessing state slices
- **Middleware:** Intercepting actions

### Basic Example
```typescript
// Store
const store = createStore(rootReducer);

// Actions
const incrementAction = { type: 'INCREMENT', payload: 5 };

// Reducer
const reducer = (state = 0, action) => {
  if (action.type === 'INCREMENT') {
    return state + action.payload;
  }
  return state;
};

// Connect to component
const count = useSelector(state => state.count);
const dispatch = useDispatch();
dispatch(incrementAction);
```

### When to Use Redux
✅ **Good for:**
- Large applications
- Complex state flows
- Multiple developers
- Time-travel debugging needed
- Middleware/interceptors needed

❌ **Bad for:**
- Simple apps
- Learning React
- Rapid prototyping

### Redux Tools
- **Redux Toolkit:** Simplified Redux setup
- **Immer:** Immutable updates
- **Redux DevTools:** Browser extension for debugging

## Zustand Pattern

**Lightweight** and simple state management (recommended for most projects).

### Key Concepts
- Minimal boilerplate
- Hooks-based API
- No providers needed (optional)
- Immer built-in

### Example
```typescript
import create from 'zustand';

const useStore = create((set) => ({
  count: 0,
  increment: () => set((state) => ({ count: state.count + 1 })),
  decrement: () => set((state) => ({ count: state.count - 1 })),
}));

// Usage
function Counter() {
  const { count, increment, decrement } = useStore();
  return (
    <>
      <p>Count: {count}</p>
      <button onClick={increment}>+</button>
      <button onClick={decrement}>-</button>
    </>
  );
}
```

### When to Use Zustand
✅ **Good for:**
- Small to medium apps
- Simplicity over features
- Global state
- Fast prototyping
- Teams preferring hooks

❌ **Bad for:**
- Very complex state
- Redux middleware needed
- Heavy DevTools usage

## Jotai Pattern

**Atomic** state management (bottom-up approach).

### Key Concepts
- Atoms: Smallest unit of state
- Composable atoms
- Minimal re-renders
- No provider overhead (optional)

### Example
```typescript
import { atom, useAtom } from 'jotai';

// Define atoms
const countAtom = atom(0);
const doubledAtom = atom(
  (get) => get(countAtom) * 2,
  (get, set, newValue) => set(countAtom, newValue)
);

// Usage
function Counter() {
  const [count, setCount] = useAtom(countAtom);
  const [doubled] = useAtom(doubledAtom);

  return (
    <>
      <p>Count: {count}</p>
      <p>Doubled: {doubled}</p>
      <button onClick={() => setCount(c => c + 1)}>Increment</button>
    </>
  );
}
```

### When to Use Jotai
✅ **Good for:**
- Fine-grained reactivity
- Atoms-based architecture
- Minimal re-renders
- Composable state

❌ **Bad for:**
- Redux-style flow required
- Traditional reducers needed

## XState (State Machines)

**Explicit** state management with state machines.

### Key Concepts
- States: Defined, enumerated states
- Transitions: How to move between states
- Context: Data stored in machine
- Actions: Effects when transitioning

### Example
```typescript
import { createMachine, useMachine } from 'xstate';

const toggleMachine = createMachine({
  initial: 'off',
  states: {
    off: {
      on: { TOGGLE: 'on' }
    },
    on: {
      on: { TOGGLE: 'off' }
    }
  }
});

function Toggle() {
  const [state, send] = useMachine(toggleMachine);

  return (
    <>
      <p>State: {state.value}</p>
      <button onClick={() => send('TOGGLE')}>
        Toggle
      </button>
    </>
  );
}
```

### When to Use XState
✅ **Good for:**
- Complex workflows
- Explicit state flows
- UI state machines
- Preventing invalid states

❌ **Bad for:**
- Simple state
- Learning state management
- Minimal overhead needed

## TanStack Query

**Server state** management (perfect for API data).

### Key Concepts
- Queries: Fetching data
- Mutations: Modifying data
- Caching: Automatic caching
- Synchronization: Keep UI in sync

### Example
```typescript
import { useQuery, useMutation } from '@tanstack/react-query';

function UsersList() {
  const { data: users, isLoading } = useQuery({
    queryKey: ['users'],
    queryFn: () => api.getUsers(),
  });

  const mutation = useMutation({
    mutationFn: (newUser) => api.createUser(newUser),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ['users'] });
    },
  });

  if (isLoading) return <div>Loading...</div>;

  return (
    <>
      <ul>
        {users?.map(u => <li key={u.id}>{u.name}</li>)}
      </ul>
      <button onClick={() => mutation.mutate({ name: 'New User' })}>
        Add User
      </button>
    </>
  );
}
```

### When to Use TanStack Query
✅ **Good for:**
- Server state management
- API data caching
- Background synchronization
- Complex data flows

| Feature | Redux | Zustand | Jotai | XState | TanStack Query |

## Decision Tree

```
Choose State Management

├─ Is it app state (user, theme)?
│  └─ useState + useContext (simple)
│  └─ Zustand (scalable)
│  └─ Jotai (fine-grained)
│
├─ Is it server state (API data)?
│  └─ TanStack Query (recommended)
│  └─ useEffect + useState (simple)
│
├─ Is it complex logic?
│  └─ XState (explicit flows)
│  └─ Redux (proven patterns)
│
└─ Is it simple global state?
   └─ useContext (if small)
   └─ Zustand (if scalable)
```

## Recommended Combinations

### Small Project
```
useState + useContext
```

### Medium Project
```
Zustand (global state) + TanStack Query (server state)
```

### Large Project
```
Redux (complex app state) + TanStack Query (server state)
+ XState (complex UI flows)
```

### Enterprise
```
Redux (standardization) + TanStack Query (data)
+ XState (workflows) + Custom middleware
```

## Migration Path

```
1. Start: useState + useContext
    ↓
2. Grow: Add Zustand if global state grows
    ↓
3. Mature: Add TanStack Query for API
    ↓
4. Complex: Add XState for workflows
    ↓
5. Enterprise: Consider Redux if needed
```

## Server vs Client State

### Server State (API data)
- Cached on server
- Can get stale
- Requires refetching
- **Tool:** TanStack Query ⭐

### Client State (app state)
- Only in browser
- Never stale
- User-controlled
- **Tool:** Zustand, Context, Redux

### UI State (local state)
- Toggle, forms, modals
- Transient
- Component-level
- **Tool:** useState ✅

---

## Best Practices

✅ **Do:**
- Keep server state separate from client state
- Use TanStack Query for API data
- Use Zustand for app state
- Use useState for local UI state
- Use XState for complex flows

❌ **Don't:**
- Mix server and client state
- Store API responses directly (normalize them)
- Overuse global state
- Make everything a global atom/action
