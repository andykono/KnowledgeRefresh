# Architectural Patterns

[[2. Architectural Patterns.md]]

Архитектурные паттерны определяют **общую структуру приложения** на высоком уровне. В отличие от дизайн-паттернов (которые решают локальные проблемы), архитектурные паттерны влияют на всю организацию кода.

---

## 1. Component-Based Architecture (React Approach)

**Основная идея:** приложение — это дерево переиспользуемых компонентов, каждый инкапсулирует свою логику и UI.

### Структура компонента в React

```typescript
// components/Button.tsx
interface ButtonProps {
  label: string;
  onClick: () => void;
  variant?: 'primary' | 'secondary';
  disabled?: boolean;
}

export function Button({
  label,
  onClick,
  variant = 'primary',
  disabled = false,
}: ButtonProps) {
  return (
    <button
      className={`btn btn-${variant}`}
      onClick={onClick}
      disabled={disabled}
    >
      {label}
    </button>
  );
}

// components/Card.tsx
interface CardProps {
  title: string;
  children: React.ReactNode;
}

export function Card({ title, children }: CardProps) {
  return (
    <div className="card">
      <h3>{title}</h3>
      {children}
    </div>
  );
}

// Composition: combine components
function UserProfile() {
  return (
    <Card title="User Profile">
      <Button label="Edit" onClick={() => {}} variant="primary" />
      <Button label="Delete" onClick={() => {}} variant="secondary" />
    </Card>
  );
}
```

### Typical App Structure

```
src/
├── components/          # Presentational components
│   ├── Button.tsx
│   ├── Card.tsx
│   └── Form.tsx
├── features/            # Feature-specific components
│   ├── Auth/
│   │   ├── LoginForm.tsx
│   │   ├── useAuth.ts
│   │   └── AuthContext.tsx
│   └── Users/
│       ├── UserList.tsx
│       ├── UserDetails.tsx
│       └── useUsers.ts
├── hooks/              # Custom hooks (logic)
│   ├── useAsync.ts
│   ├── useAsync.ts
│   └── useLocalStorage.ts
├── services/           # Business logic
│   ├── apiClient.ts
│   └── authService.ts
├── App.tsx             # Root component
└── index.tsx
```

### Advantages & Disadvantages

✅ **Pro:**
- Highly reusable components
- Easy to test isolated components
- Clear hierarchy

❌ **Con:**
- Prop drilling (pass props through many levels)
- Can lead to shallow component trees
- State management can get complex

## 2. MVVM (Model-View-ViewModel)

**Основная идея:** отделить UI (View) от логики (ViewModel) через двусторонний binding.

### Traditional MVVM (Angular)

```typescript
// user.model.ts
export interface User {
  id: number;
  name: string;
  email: string;
}

// user.viewmodel.ts
import { Injectable } from '@angular/core';
import { BehaviorSubject, Observable } from 'rxjs';
import { UserService } from './user.service';

@Injectable()
export class UserViewModel {
  private userSubject = new BehaviorSubject<User | null>(null);
  public user$: Observable<User> = this.userSubject.asObservable();

  constructor(private userService: UserService) {}

  loadUser(id: number): void {
    this.userService.getUser(id).subscribe(user => {
      this.userSubject.next(user);
    });
  }

  updateUser(user: User): void {
    this.userService.updateUser(user).subscribe(updated => {
      this.userSubject.next(updated);
    });
  }
}

// user.component.ts
@Component({
  selector: 'app-user',
  template: `
    <div *ngIf="vm.user$ | async as user">
      <h1>{{ user.name }}</h1>
      <p>{{ user.email }}</p>
      <button (click)="vm.updateUser(user)">Save</button>
    </div>
  `,
})
export class UserComponent {
  constructor(public vm: UserViewModel) {
    vm.loadUser(1);
  }
}
```

### React Version (Hooks)

```typescript
// types/User.ts
export interface User {
  id: number;
  name: string;
  email: string;
}

// viewmodels/useUserViewModel.ts
import { useState, useEffect, useCallback } from 'react';
import { userService } from '../services/userService';

export function useUserViewModel(userId: number) {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    setLoading(true);
    userService
      .getUser(userId)
      .then(setUser)
      .catch(setError)
      .finally(() => setLoading(false));
  }, [userId]);

  const updateUser = useCallback(async (updated: User) => {
    try {
      const result = await userService.updateUser(updated);
      setUser(result);
    } catch (err) {
      setError(err instanceof Error ? err : new Error('Update failed'));
    }
  }, []);

  return { user, loading, error, updateUser };
}

// components/UserProfile.tsx
function UserProfile({ userId }: { userId: number }) {
  const { user, loading, error, updateUser } = useUserViewModel(userId);

  if (loading) return <div>Loading...</div>;
  if (error) return <div>Error: {error.message}</div>;
  if (!user) return <div>No user found</div>;

  return (
    <div>
      <h1>{user.name}</h1>
      <p>{user.email}</p>
      <button onClick={() => updateUser({ ...user, name: 'Updated' })}>
        Save
      </button>
    </div>
  );
}
```

### Key Characteristics

- **Model:** Data structure and business logic
- **View:** UI presentation (dumb component)
- **ViewModel:** Binds View to Model, handles user interactions

## 3. MVC (Model-View-Controller)

**Основная идея:** разделить приложение на три слоя: Model (данные), View (UI), Controller (логика).

Менее популярен в React, но вот как это выглядит в контексте:

```typescript
// models/UserModel.ts
export interface User {
  id: number;
  name: string;
  email: string;
}

export class UserModel {
  private users: User[] = [];

  addUser(user: User): void {
    this.users.push(user);
  }

  getUser(id: number): User | undefined {
    return this.users.find(u => u.id === id);
  }

  getAllUsers(): User[] {
    return [...this.users];
  }

  deleteUser(id: number): void {
    this.users = this.users.filter(u => u.id !== id);
  }
}

// controllers/UserController.ts
export class UserController {
  constructor(private model: UserModel) {}

  createUser(name: string, email: string): User {
    const user: User = {
      id: Date.now(),
      name,
      email,
    };
    this.model.addUser(user);
    return user;
  }

  getUser(id: number): User | undefined {
    return this.model.getUser(id);
  }

  getAllUsers(): User[] {
    return this.model.getAllUsers();
  }

  deleteUser(id: number): void {
    this.model.deleteUser(id);
  }
}

// components/UserListView.tsx
interface UserListViewProps {
  users: User[];
  onAddUser: (name: string, email: string) => void;
  onDeleteUser: (id: number) => void;
}

function UserListView({ users, onAddUser, onDeleteUser }: UserListViewProps) {
  return (
    <div>
      <h1>Users</h1>
      <ul>
        {users.map(user => (
          <li key={user.id}>
            {user.name} ({user.email})
            <button onClick={() => onDeleteUser(user.id)}>Delete</button>
          </li>
        ))}
      </ul>
      <form onSubmit={(e) => {
        e.preventDefault();
        const formData = new FormData(e.currentTarget);
        onAddUser(
          formData.get('name') as string,
          formData.get('email') as string
        );
      }}>
        <input name="name" placeholder="Name" required />
        <input name="email" type="email" placeholder="Email" required />
        <button type="submit">Add User</button>
      </form>
    </div>
  );
}

// Container component (connects View with Controller)
function UserListContainer() {
  const [controller] = useState(() => new UserController(new UserModel()));
  const [users, setUsers] = useState<User[]>([]);

  const handleAddUser = (name: string, email: string) => {
    controller.createUser(name, email);
    setUsers(controller.getAllUsers());
  };

  const handleDeleteUser = (id: number) => {
    controller.deleteUser(id);
    setUsers(controller.getAllUsers());
  };

  useEffect(() => {
    setUsers(controller.getAllUsers());
  }, [controller]);

  return (
    <UserListView
      users={users}
      onAddUser={handleAddUser}
      onDeleteUser={handleDeleteUser}
    />
  );
}
```

## 4. Clean Architecture

**Основная идея:** слои с четкими границами, где внутренние слои не зависят от внешних.

```
entities/          # Core business rules, interfaces
├── User.ts
└── Order.ts

usecases/          # Application business rules
├── CreateUser.ts
├── UpdateUser.ts
└── DeleteUser.ts

repositories/      # Data access layer
├── IUserRepository.ts
└── UserRepository.ts

presenters/        # Presentation logic
├── UserPresenter.ts
└── OrderPresenter.ts

controllers/       # Entry points
├── UserController.ts
└── OrderController.ts
```

### Example Implementation

```typescript
// entities/User.ts
export interface User {
  id: string;
  name: string;
  email: string;
}

// repositories/IUserRepository.ts
export interface IUserRepository {
  save(user: User): Promise<void>;
  getById(id: string): Promise<User | null>;
  getAll(): Promise<User[]>;
  delete(id: string): Promise<void>;
}

// usecases/CreateUserUseCase.ts
export class CreateUserUseCase {
  constructor(private userRepository: IUserRepository) {}

  async execute(name: string, email: string): Promise<User> {
    const user: User = {
      id: Date.now().toString(),
      name,
      email,
    };

    await this.userRepository.save(user);
    return user;
  }
}

// repositories/UserRepository.ts (implementation)
export class UserRepository implements IUserRepository {
  constructor(private apiClient: ApiClient) {}

  async save(user: User): Promise<void> {
    await this.apiClient.post('/users', user);
  }

  async getById(id: string): Promise<User | null> {
    return this.apiClient.get(`/users/${id}`);
  }

  async getAll(): Promise<User[]> {
    return this.apiClient.get('/users');
  }

  async delete(id: string): Promise<void> {
    await this.apiClient.delete(`/users/${id}`);
  }
}

// presenters/UserPresenter.ts
export interface UserDTO {
  id: string;
  displayName: string;
  contactEmail: string;
}

export class UserPresenter {
  present(user: User): UserDTO {
    return {
      id: user.id,
      displayName: user.name.toUpperCase(),
      contactEmail: user.email,
    };
  }
}

// controllers/UserController.ts
export class UserController {
  constructor(
    private createUserUseCase: CreateUserUseCase,
    private userPresenter: UserPresenter
  ) {}

  async create(name: string, email: string): Promise<UserDTO> {
    const user = await this.createUserUseCase.execute(name, email);
    return this.userPresenter.present(user);
  }
}

// React integration
function UserCreateForm() {
  const controller = useRef(
    new UserController(
      new CreateUserUseCase(
        new UserRepository(apiClient)
      ),
      new UserPresenter()
    )
  ).current;

  const [loading, setLoading] = useState(false);

  const handleSubmit = async (e: React.FormEvent<HTMLFormElement>) => {
    e.preventDefault();
    const formData = new FormData(e.currentTarget);
    setLoading(true);

    try {
      const result = await controller.create(
        formData.get('name') as string,
        formData.get('email') as string
      );
      console.log('Created:', result);
    } finally {
      setLoading(false);
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      <input name="name" placeholder="Name" required />
      <input name="email" type="email" placeholder="Email" required />
      <button disabled={loading}>Create User</button>
    </form>
  );
}
```

## 5. Onion Architecture (Луковая архитектура)

**Основная идея:** слои расположены как луковица, где внутренние слои независимы от внешних. Инверсия зависимостей направлена к центру.

### Слои Onion Architecture

```
┌─────────────────────────────────────────┐
│        Presentation Layer (UI)          │  ← Зависит от остального
├─────────────────────────────────────────┤
│        Application Services Layer       │  ← Координирует использованные casos
├─────────────────────────────────────────┤
│        Domain Services Layer            │  ← Бизнес-логика
├─────────────────────────────────────────┤
│        Domain Model (Entities)          │  ← Ядро, ничего не знает о внешнем
└─────────────────────────────────────────┘
        Infrastructure Services
        (DB, API, File System)
```

### Структура папок

```
src/
├── core/                          # Ядро (Domain Model)
│   ├── entities/
│   │   ├── User.ts
│   │   └── Order.ts
│   └── interfaces/                # Contracts (interfaces)
│       ├── IUserRepository.ts
│       └── IOrderService.ts
│
├── domain/                        # Domain Services Layer
│   ├── services/
│   │   ├── UserDomainService.ts
│   │   └── OrderDomainService.ts
│   └── specifications/            # Domain logic (predicates)
│       └── ActiveUserSpec.ts
│
├── application/                   # Application Services Layer
│   ├── services/
│   │   ├── CreateUserService.ts
│   │   └── PlaceOrderService.ts
│   └── dto/                       # Data Transfer Objects
│       ├── CreateUserDTO.ts
│       └── UserResponseDTO.ts
│
├── presentation/                  # Presentation Layer (UI)
│   ├── components/
│   │   ├── UserForm.tsx
│   │   └── OrderList.tsx
│   ├── pages/
│   │   └── UserPage.tsx
│   └── controllers/              # API endpoints
│       ├── UserController.ts
│       └── OrderController.ts
│
└── infrastructure/               # Infrastructure (outside)
    ├── persistence/
    │   ├── UserRepository.ts      # Implements IUserRepository
    │   └── OrderRepository.ts
    ├── external/
    │   └── PaymentGateway.ts      # 3rd party integrations
    └── config/
        └── Database.ts
```

### React Example (Onion Architecture)

```typescript
// core/entities/User.ts
export interface User {
  id: string;
  name: string;
  email: string;
  isActive: boolean;
}

// core/interfaces/IUserRepository.ts
export interface IUserRepository {
  save(user: User): Promise<void>;
  getById(id: string): Promise<User | null>;
  getAllActive(): Promise<User[]>;
}

// domain/services/UserDomainService.ts
export class UserDomainService {
  constructor(private userRepository: IUserRepository) {}

  async activateUser(userId: string): Promise<User> {
    const user = await this.userRepository.getById(userId);
    if (!user) throw new Error('User not found');

    // Pure domain logic
    user.isActive = true;

    await this.userRepository.save(user);
    return user;
  }
}

// application/services/CreateUserService.ts
export class CreateUserService {
  constructor(
    private userRepository: IUserRepository,
    private userDomainService: UserDomainService
  ) {}

  async execute(name: string, email: string): Promise<User> {
    const user: User = {
      id: Date.now().toString(),
      name,
      email,
      isActive: true,
    };

    await this.userRepository.save(user);
    return user;
  }
}

// application/dto/CreateUserDTO.ts
export interface CreateUserDTO {
  name: string;
  email: string;
}

export interface UserResponseDTO {
  id: string;
  name: string;
  email: string;
  isActive: boolean;
}

// infrastructure/persistence/UserRepository.ts
export class UserRepository implements IUserRepository {
  constructor(private apiClient: ApiClient) {}

  async save(user: User): Promise<void> {
    await this.apiClient.post('/users', user);
  }

  async getById(id: string): Promise<User | null> {
    return this.apiClient.get(`/users/${id}`);
  }

  async getAllActive(): Promise<User[]> {
    return this.apiClient.get(`/users?active=true`);
  }
}

// presentation/components/UserForm.tsx
import { useState } from 'react';

interface UserFormProps {
  onSubmit: (dto: CreateUserDTO) => Promise<void>;
  loading: boolean;
}

export function UserForm({ onSubmit, loading }: UserFormProps) {
  const [name, setName] = useState('');
  const [email, setEmail] = useState('');

  const handleSubmit = async (e: React.FormEvent) => {
    e.preventDefault();
    await onSubmit({ name, email });
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        value={name}
        onChange={(e) => setName(e.target.value)}
        placeholder="Name"
      />
      <input
        value={email}
        onChange={(e) => setEmail(e.target.value)}
        placeholder="Email"
        type="email"
      />
      <button disabled={loading}>Create User</button>
    </form>
  );
}

// App setup (Dependency Injection)
const userRepository = new UserRepository(apiClient);
const userDomainService = new UserDomainService(userRepository);
const createUserService = new CreateUserService(
  userRepository,
  userDomainService
);

function App() {
  return (
    <UserForm
      onSubmit={(dto) => createUserService.execute(dto.name, dto.email)}
      loading={false}
    />
  );
}
```

### Onion Architecture vs Clean Architecture

| Аспект | Onion | Clean |
| :-- | :-- | :-- |
| **Зависимости** | Инверсия к центру | Инверсия к центру |
| **Слои** | 4 слоя | 4+ слоев |
| **Фокус** | Domain-driven | Layer-driven |
| **Complexity** | Средняя | Высокая |
| **Best For** | Domain-heavy apps | Very large enterprise |

### When to Use Onion

✅ **Good for:**
- Domain-driven design (DDD)
- Complex business logic
- Enterprise applications
- Testability important
- Team is experienced

❌ **Bad for:**
- Simple CRUD apps
- Small teams
- Rapid prototyping
- Learning architecture

## 6. Feature-Sliced Design (FSD)

**Основная идея:** группировать код по **фичам** (вертикали), а не по типам (горизонтали).

### Traditional Structure (не рекомендуется)

```
src/
├── components/
├── hooks/
├── utils/
├── services/
└── ...
```

### FSD Structure (рекомендуется)

```
src/
├── app/                    # Global setup, routing
│   ├── App.tsx
│   └── routing.tsx
│
├── shared/                 # Shared utils, constants
│   ├── api/
│   ├── hooks/
│   └── ui/
│
├── entities/               # Domain entities
│   ├── User/
│   │   ├── model/          # Types, interfaces
│   │   ├── ui/             # Components
│   │   └── hooks/
│   └── Order/
│
├── features/               # Features (user interactions)
│   ├── Auth/
│   │   ├── model/
│   │   ├── ui/
│   │   ├── api.ts
│   │   └── hooks/
│   ├── UserProfile/
│   └── OrderHistory/
│
└── widgets/                # Complex reusable components
    ├── Navbar/
    ├── Sidebar/
    └── Footer/
```

### FSD Example

```typescript
// features/Auth/model/types.ts
export interface User {
  id: string;
  email: string;
  name: string;
}

// features/Auth/model/store.ts
import { createContext, useState } from 'react';

export const AuthContext = createContext<{
  user: User | null;
  login: (email: string, password: string) => Promise<void>;
  logout: () => Promise<void>;
} | null>(null);

// features/Auth/api.ts
export const authApi = {
  login: async (email: string, password: string): Promise<User> => {
    const res = await fetch('/api/auth/login', {
      method: 'POST',
      body: JSON.stringify({ email, password }),
    });
    return res.json();
  },
  logout: async (): Promise<void> => {
    await fetch('/api/auth/logout', { method: 'POST' });
  },
};

// features/Auth/ui/LoginForm.tsx
import { useState } from 'react';
import { authApi } from '../api';

interface LoginFormProps {
  onSuccess: (user: User) => void;
}

export function LoginForm({ onSuccess }: LoginFormProps) {
  const [email, setEmail] = useState('');
  const [password, setPassword] = useState('');
  const [loading, setLoading] = useState(false);

  const handleSubmit = async (e: React.FormEvent) => {
    e.preventDefault();
    setLoading(true);
    try {
      const user = await authApi.login(email, password);
      onSuccess(user);
    } finally {
      setLoading(false);
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        type="email"
        value={email}
        onChange={(e) => setEmail(e.target.value)}
        placeholder="Email"
      />
      <input
        type="password"
        value={password}
        onChange={(e) => setPassword(e.target.value)}
        placeholder="Password"
      />
      <button disabled={loading}>Login</button>
    </form>
  );
}

// features/Auth/hooks/useAuth.ts
import { useContext } from 'react';
import { AuthContext } from '../model/store';

export function useAuth() {
  const context = useContext(AuthContext);
  if (!context) {
    throw new Error('useAuth must be used inside AuthProvider');
  }
  return context;
}

// app/App.tsx
function App() {
  return (
    <AuthProvider>
      <Router>
        <Navbar />
        <Routes>
          <Route path="/" element={<Home />} />
          <Route path="/profile" element={<Profile />} />
        </Routes>
      </Router>
    </AuthProvider>
  );
}
```

### Principles of FSD

✅ **Pro:**
- Easy to scale
- Clear separation of concerns
- Easy to find code (everything related to feature in one folder)
- Can work with features independently

❌ **Con:**
- Overkill for small projects
- Initial setup complexity
- Learning curve

---

## Comparison: Which One to Use?

| Pattern | Best For | Team Size | Complexity |
| :-- | :-- | :-- | :-- |
| **Component-Based** | Simple apps, learning React | 1-2 | Low |
| **MVVM + Hooks** | Medium apps with state | 3-5 | Medium |
| **MVC** | Enterprise legacy apps | Large | High |
| **Clean Architecture** | Testable, enterprise code | 5+ | High |
| **Onion Architecture** | Domain-driven, complex logic | 5+ | High |
| **FSD** | Scalable, modular apps | 5+ | Medium-High |

---

## Recommended for React

1. **Start:** Component-Based Architecture
2. **Grow:** Add MVVM with hooks and Context API
3. **Scale:** Introduce FSD for feature organization (⭐ **Recommended**)
4. **Enterprise:** Add Clean Architecture OR Onion Architecture principles

**Note:** FSD is the most suitable for React. Clean/Onion architectures are better for heavy domain-driven applications with complex business logic.

## SOLID Principles (Foundation for All)

These principles underpin good architecture:

**S**ingle Responsibility — each module does one thing
**O**pen/Closed — open for extension, closed for modification
**L**iskov Substitution — subtypes can replace base types
**I**nterface Segregation — don't depend on unused interfaces
**D**ependency Inversion — depend on abstractions, not concrete classes

```typescript
// ❌ Bad (violates SRP and DIP)
class UserService {
  saveToDatabase(user: User) { /* ... */ }
  sendEmail(email: string) { /* ... */ }
  logActivity(action: string) { /* ... */ }
}

// ✅ Good (SRP + DIP)
interface IUserRepository {
  save(user: User): Promise<void>;
}

interface IEmailService {
  send(email: string): Promise<void>;
}

class UserService {
  constructor(
    private userRepository: IUserRepository,
    private emailService: IEmailService
  ) {}
```
