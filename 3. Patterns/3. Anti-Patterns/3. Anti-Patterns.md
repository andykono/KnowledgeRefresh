# Anti-Patterns: What NOT to Do

[[../README.md]]

Антипаттерны — это распространённые **ошибки и плохие практики**, которые приводят к проблемам. Изучите их, чтобы избежать.

---

## What Are Anti-Patterns?

Антипаттерны:
- ❌ Выглядят как правильное решение
- ❌ Но приводят к проблемам: баги, утечки памяти, сложность
- ❌ Часто встречаются в коде
- ✅ Легко исправить, если знать

---

## Categories of Anti-Patterns

### [[3.1 React Anti-Patterns]]
Most common mistakes in React:
- Prop drilling (without Context)
- Missing dependencies
- Memory leaks
- Mutable state
- Callback hell

### [[3.2 State Management Anti-Patterns]]
Wrong ways to manage state:
- Global mutable state
- Overly complex reducers
- Circular dependencies
- Over-fetching

### [[3.3 TypeScript Anti-Patterns]]
TypeScript mistakes:
- `any` type overuse
- Wrong type inference
- Breaking type safety
- Not using generics

Before committing code:

```typescript
// ❌ Missing dependency

const user = state.user;
```

## Most Common Anti-Patterns

### 1. Prop Drilling
```typescript
// ❌ Bad: Pass through multiple levels
<Component user={user}>
  <Child user={user}>
    <GrandChild user={user} />
  </Child>
</Component>

// ✅ Good: Use Context
<UserProvider value={user}>
  <Component>
    <Child>
      <GrandChild /> {/* access via useUser() */}
    </Child>
  </Component>
</UserProvider>
```

### 2. Missing useEffect Dependencies
```typescript
// ❌ Bad: userId not in dependency array
useEffect(() => {
  fetchUser(userId);
}, []); // Will fetch OLD userId!

// ✅ Good
useEffect(() => {
  fetchUser(userId);
}, [userId]); // Refetch when userId changes
```

### 3. Mutable State
```typescript
// ❌ Bad: Mutating state directly
const user = state.user;
user.name = 'John';
setState(state); // React won't detect change

// ✅ Good: Create new object
const user = { ...state.user, name: 'John' };
setState({ ...state, user });
```

### 4. Memory Leaks
```typescript
// ❌ Bad: No cleanup
useEffect(() => {
  const timer = setInterval(() => {
    // Timer keeps running after unmount!
  }, 1000);
}, []);

// ✅ Good: Cleanup function
useEffect(() => {
  const timer = setInterval(() => {
    // ...
  }, 1000);
  return () => clearInterval(timer); // Cleanup
}, []);
```

### 5. Callback Hell
```typescript
// ❌ Bad: Nested callbacks
const handleClick = useCallback(() => {
  const handleChange = useCallback(() => {
    const handleSubmit = useCallback(() => {
      // Hell!
    }, [deps]);
  }, [deps]);
}, [deps]);

// ✅ Good: Separate functions
const handleSubmit = useCallback(() => {
  // ...
}, [deps]);

const handleChange = useCallback(() => {
  // ...
}, [deps]);

const handleClick = useCallback(() => {
  // ...
}, [deps]);
```

### 6. Recreating Objects/Functions
```typescript
// ❌ Bad: Function recreated every render
<Child onClick={() => handleClick()} />

// ✅ Good: Memoized or stable
const onClick = useCallback(() => handleClick(), [handleClick]);
<Child onClick={onClick} />
```

### 7. Using useState When useReducer Needed
```typescript
// ❌ Bad: Too many related states
const [name, setName] = useState('');
const [email, setEmail] = useState('');
const [password, setPassword] = useState('');
const [error, setError] = useState('');
const [loading, setLoading] = useState(false);

// ✅ Good: useReducer for related state
const [state, dispatch] = useReducer(reducer, initialState);
```

### 8. Console.log Debugging
```typescript
// ❌ Bad: console.log left in code
useEffect(() => {
  console.log('user:', user);
  console.log('loading:', loading);
  // ...
}, [user]);

// ✅ Good: Use React DevTools debugger
useEffect(() => {
  debugger; // Click breakpoint in DevTools
  // ...
}, [user]);
```

### 9. "Temporary" Code
```typescript
// ❌ Bad: Temporary code that stays
if (process.env.NODE_ENV === 'development') {
  // This was "temporary" 2 years ago
  // Now it's production code
}

// ✅ Good: Remove or document properly
// Use feature flags instead
if (featureFlags.newUI) {
  return <NewUI />;
}
```

### 10. Comments Explaining Bad Code
```typescript
// ❌ Bad: Comment explaining why code is bad
// This is a hack because X, we should refactor to Y
const weirdCalculation = x => x * 2 - 1 + 3;

// ✅ Good: Just refactor!
const calculateValue = (x: number) => x * 2 + 2;
```

## Red Flags in Code Review

When reviewing code, watch for:

```typescript
// 🚩 Empty dependency array when using external variables
useEffect(() => {
  console.log(someVar); // But someVar not in deps
}, []);

// 🚩 Undefined checks on values that should always exist
if (props.user !== undefined) { // If user is required, it's never undefined
  // ...
}

// 🚩 Multiple similar functions that could be one
const handleAddUser = () => { /* ... */ }
const handleAddPost = () => { /* ... */ }
const handleAddComment = () => { /* ... */ }

// 🚩 useCallback with no dependencies
const handleClick = useCallback(() => {
  // If nothing in deps, just use regular function
  // ...
}, []);

// 🚩 Props named like internal state
<Component
  value={userValue}  // Looks like internal state
  onChange={setUserValue}  // onChange suggests controlled
/>

// 🚩 Spread props without knowing what's inside
<Component {...props} /> // What props are being passed?

// 🚩 Boolean props with long names
<Component isUserLoggedInAndHasPermissionToEditProfile={true} />

// 🚩 Magic numbers without explanation
setTimeout(() => { /* ... */ }, 500); // Why 500?
```

## Performance Anti-Patterns

### Unnecessary Re-renders
```typescript
// ❌ Bad: Object created in props (new reference each render)
<Child data={{ id: 1, name: 'John' }} />

// ✅ Good: Memoize or extract outside
const data = useMemo(() => ({ id: 1, name: 'John' }), []);
<Child data={data} />
```

### Missing React.memo
```typescript
// ❌ Bad: Heavy component re-renders unnecessarily
function HeavyComponent({ value }: Props) {
  return <ExpensiveCalculation value={value} />;
}

// ✅ Good: Memoize if props don't change
const HeavyComponent = React.memo(({ value }: Props) => {
  return <ExpensiveCalculation value={value} />;
});
```

## Debugging Anti-Patterns

**Tools to catch them:**
- ESLint plugin: `eslint-plugin-react-hooks`
- TypeScript strict mode
- React DevTools Profiler
- React Strict Mode (catches side effects)

```typescript
// Enable Strict Mode to catch issues
root.render(
  <React.StrictMode>
    <App />
  </React.StrictMode>
);
```

## Refactoring Checklist

When refactoring code:

- [ ] Remove console.log statements
- [ ] Check all useEffect dependencies
- [ ] Look for memory leaks
- [ ] Simplify unnecessary nesting
- [ ] Replace props drilling with Context
- [ ] Extract complex logic to hooks
- [ ] Add proper TypeScript types
- [ ] Check for mutable state
- [ ] Simplify callbacks
- [ ] Remove "temporary" code
