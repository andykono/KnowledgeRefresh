# Design Patterns: Comprehensive Guide

[[../README.md]]

Паттерн программирования — это **переиспользуемое решение** типовой задачи проектирования: не код, а идея, которую можно адаптировать под любой язык или фреймворк. Паттерны помогают писать чистый, масштабируемый и понятный код.[^1_1][^1_2]

Design patterns are proven solutions to common problems in software design. They're language-agnostic blueprints that guide you toward maintainable, scalable architectures. This guide focuses on **React-oriented examples**, bridging traditional OOP patterns with modern functional/hook-based approaches.

---

## Types of Design Patterns

Паттерны делятся на **4 основные категории**:

### 1. **Creational Patterns** (Порождающие)
**Цель:** управлять **созданием объектов**

Решают проблемы: как создавать объекты, управлять инициализацией, скрывать сложность создания.

| Pattern | Problem | React Example |
| :-- | :-- | :-- |
| [[1.1 Singleton]] | Один глобальный инстанс | API клиент, auth service |
| [[1.2 Factory Method]] | Гибкое создание разных типов объектов | Dynamic components by metadata |
| Abstract Factory | Создание семейств объектов | Theme providers, adapter systems |
| Builder | Сложное поэтапное конструирование | Form builders, query builders |
| Prototype | Клонирование объектов | State cloning, undo/redo |

### 2. **Structural Patterns** (Структурные)
**Цель:** управлять **отношениями между объектами** и их комбинированием

Решают проблемы: как组織 классы, как добавлять функциональность без изменения кода, как упростить сложные интерфейсы.

| Pattern | Problem | React Example |
| :-- | :-- | :-- |
| [[1.6 Facade]] | Упрощение сложной подсистемы | Service layer, API wrapper |
| [[1.4 Decorator]] | Динамическое добавление поведения | HOCs (withAuth, withTheme) |
| Adapter | Совместимость несовместимых интерфейсов | API response transformers |
| Bridge | Разделение абстракции от реализации | Platform-specific renderers |
| Composite | Работа с деревьями объектов | Menu hierarchies, DOM-like trees |
| Proxy | Замена объекта на суррогат | Lazy loading, access control |
| Flyweight | Оптимизация памяти (шеринг объектов) | Pooling, memoization |

### 3. **Behavioral Patterns** (Поведенческие)
**Цель:** управлять **взаимодействием и распределением ответственности**

Решают проблемы: как организовать общение между объектами, как распределить ответственность, как упростить логику.

| Pattern | Problem | React Example |
| :-- | :-- | :-- |
| [[1.3 Observer]] | Подписка на события/состояние | Custom hooks, Context API |
| [[1.5 Strategy]] | Выбор алгоритма на лету | Sorting, filtering, validation rules |
| Command | Инкапсуляция команд как объектов | Action creators, undo/redo |
| Chain of Responsibility | Последовательная обработка запроса | Middleware, error boundaries |
| Iterator | Последовательный доступ к элементам | Generators, lazy evaluation |
| Mediator | Централизованное взаимодействие | Redux, state manager |
| State | Изменение поведения при смене состояния | State machines, xstate |
| Template Method | Скелет алгоритма в базовом классе | Hooks abstraction pattern |
| Visitor | Добавление операций без изменения классов | AST traversal, tree processing |
| Memento | Сохранение и восстановление состояния | Undo/redo, session persistence |
| Interpreter | Определение грамматики и интерпретация | Query builders, expression parsers |

### 4. **Architectural Patterns** (Архитектурные)
**Цель:** определить **общую структуру приложения**

Решают проблемы: как организовать код на высоком уровне, как разделить ответственность между слоями.

| Pattern | Approach | React Use Case |
| :-- | :-- | :-- |
| [[2.1 MVC, MVVM, Architectural Patterns\|Component-based]] | Компоненты как основная единица | React components with hooks |
| [[2.1 MVC, MVVM, Architectural Patterns\|MVVM]] | Model ↔ ViewModel ↔ View binding | React with state management |
| [[2.1 MVC, MVVM, Architectural Patterns\|Clean Architecture]] | Слои: Entity → Use Case → Interface | Feature-based folder structure |
| [[2.1 MVC, MVVM, Architectural Patterns\|Feature-Sliced Design (FSD)]] | Горизонтальные слои + вертикальные фичи | Best practice for scalability |

## Key Differences Between Types

| Aspect | Creational | Structural | Behavioral | Architectural |
| :-- | :-- | :-- | :-- | :-- |
| **Масштаб** | Отдельные объекты | Комбинирование объектов | Взаимодействие объектов | Цело приложение |
| **Фокус** | **Как создавать** | **Как комбинировать** | **Как взаимодействовать** | **Как организовать** |
| **Трудность** | Низкая | Средняя | Средняя | Высокая |
| **Времена применения** | При создании объектов | При дизайне классов | При взаимодействии | При планировании проекта |

## How to Use This Guide

1. **Identify the problem** — какую задачу вы решаете? Создание объектов? Комбинирование? Взаимодействие?
2. **Pick the right category** — выберите категорию выше
3. **Choose the pattern** — найдите подходящий паттерн
4. **Adapt to React** — все примеры здесь адаптированы под React + TypeScript

## Common Mistakes

- ❌ Overusing Singletons (приводит к tight coupling)
- ❌ Missing Composition (вместо Inheritance используйте Composition)
- ❌ Observer pattern без unsubscribe (утечки памяти)
- ❌ Nested HOCs (HOC hell вместо Composition/Hooks)
- ❌ Ignoring Architectural Patterns (спагетти-код растет)

## 1. Singleton

Одна-единственная глобальная сущность: логгер, конфиг, storage-клиент и т.п.[^1_3][^1_4]
Идея: класс можно инстанцировать только один раз, остальное — ссылки на тот же объект.

```ts
class ApiClient {
  private static instance: ApiClient | null = null;

  private constructor(private baseUrl: string) {}

  static getInstance() {
    if (!ApiClient.instance) {
      ApiClient.instance = new ApiClient("https://api.example.com");
    }
    return ApiClient.instance;
  }

  get(path: string) {
    return fetch(this.baseUrl + path);
  }
}

const api1 = ApiClient.getInstance();
const api2 = ApiClient.getInstance();
console.log(api1 === api2); // true
```

Где применить во фронте: клиент API, глобальный event-bus, адаптер к локальному хранилищу.

## 2. Factory (Простая фабрика / Factory Method)

Паттерн для создания объектов, когда логика выбора конкретной реализации не должна быть разбросана по коду.[^1_4][^1_5]
Вместо `new SomeClass()` везде, у тебя есть **фабрика**, которая решает, что именно создать.

```ts
type ChartType = "line" | "bar";

interface Chart {
  render(container: HTMLElement): void;
}

class LineChart implements Chart { /* ... */ render() {} }
class BarChart implements Chart { /* ... */ render() {} }

class ChartFactory {
  static create(type: ChartType): Chart {
    switch (type) {
      case "line": return new LineChart();
      case "bar":  return new BarChart();
      default:     throw new Error("Unknown chart type");
    }
  }
}

const chart = ChartFactory.create("line");
chart.render(document.getElementById("root")!);
```

Во фронте: фабрики форматов (CSV/JSON/XML), адаптеры под разные API, создание компонентов по метаданным.

## 3. Observer (Наблюдатель)

Есть один объект-**subject** и много **observer**; при изменении subject он уведомляет всех подписчиков.[^1_6][^1_7]
По сути — основа event-driven подхода и реактивности.

```ts
type Listener<T> = (value: T) => void;

class Observable<T> {
  private listeners = new Set<Listener<T>>();

  subscribe(listener: Listener<T>) {
    this.listeners.add(listener);
    return () => this.listeners.delete(listener); // unsubscribe
  }

  next(value: T) {
    this.listeners.forEach(l => l(value));
  }
}

const counter$ = new Observable<number>();

const unsubscribe = counter$.subscribe(v => console.log("Counter:", v));
counter$.next(1);
counter$.next(2);
unsubscribe();
counter$.next(3); // уже не логает
```

Во фронте: RxJS в Angular, event bus, собственные хранилища состояния, веб-сокеты.

## 4. Strategy

У тебя есть алгоритм, который можно менять “на лету”, не трогая контекст.[^1_8][^1_4]
Примеры: разные способы сортировки, разные политики кеширования, разные способы валидации.

```ts
type SortStrategy<T> = (items: T[]) => T[];

const sortAsc: SortStrategy<number> = arr => [...arr].sort((a, b) => a - b);
const sortDesc: SortStrategy<number> = arr => [...arr].sort((a, b) => b - a);

class ListView {
  constructor(private sortStrategy: SortStrategy<number>) {}

  setSortStrategy(strategy: SortStrategy<number>) {
    this.sortStrategy = strategy;
  }

  render(data: number[]) {
    const sorted = this.sortStrategy(data);
    console.log(sorted);
  }
}

const list = new ListView(sortAsc);
list.render([3, 1, 2]);   // [1, 2, 3]
list.setSortStrategy(sortDesc);
list.render([3, 1, 2]);   // [3, 2, 1]
```

Во фронте: выбор алгоритма фильтрации/поиска, разные layout-стратегии, разные политики ретрая запросов.

## 5. Decorator

Добавляет поведение объекту **без изменения** его класса, оборачивая его другим объектом.[^1_4][^1_8]
В JS/TS это часто “обвязка” над функцией или классом.

```ts
interface HttpClient {
  get(url: string): Promise<Response>;
}

class FetchClient implements HttpClient {
  get(url: string) { return fetch(url); }
}

// Декоратор-логгер
class LoggingClient implements HttpClient {
  constructor(private inner: HttpClient) {}

  async get(url: string) {
    console.log("[HTTP] GET", url);
    const res = await this.inner.get(url);
    console.log("[HTTP] status", res.status);
    return res;
  }
}

const client = new LoggingClient(new FetchClient());
client.get("/users");
```

Во фронте: логирование запросов, добавление кэша, retry/backoff, метрики вокруг существующих сервисов.

## 6. Facade

**Фасад** прячет сложный внутренний мир за простым API.[^1_8][^1_4]
Ты делаешь один сервис/класс, который объединяет несколько подсистем.

```ts
class AuthService {
  constructor(
    private api: ApiClient,
    private storage: Storage,
  ) {}

  async login(email: string, password: string) {
    const res = await this.api.post("/login", { email, password });
    const { token, user } = await res.json();
    this.storage.setItem("token", token);
    return user;
  }

  logout() {
    this.storage.removeItem("token");
  }

  getToken() {
    return this.storage.getItem("token");
  }
}
```

Во фронте: фасад над “зоопарком” API, localStorage/IndexedDB, несколькими состояниями (например, Facade сервис в Angular).

## 7. MVC / MVVM / Component-based

Высокоуровневые архитектурные паттерны для разделения UI и логики.[^1_9][^1_10]

- **MVC**: Model–View–Controller — историческая база; Angular по духу ближе к этому подходу.[^1_10]
- **MVVM**: Model–View–ViewModel — биндинги между View и ViewModel; похож на Angular с two-way binding и React с hooks.[^1_2]
- **Component-based**: как в React/Angular — приложение как дерево компонент, каждый инкапсулирует состояние и UI.[^1_11][^1_9]

Простейший пример MVVM-подхода в React (упрощённо):

```tsx
// ViewModel
function useCounterViewModel() {
  const [count, setCount] = useState(0);
  const inc = () => setCount(c => c + 1);
  const dec = () => setCount(c => c - 1);
  return { count, inc, dec };
}

// View
function Counter() {
  const vm = useCounterViewModel();
  return (
    <div>
      <button onClick={vm.dec}>-</button>
      <span>{vm.count}</span>
      <button onClick={vm.inc}>+</button>
    </div>
  );
}
```
