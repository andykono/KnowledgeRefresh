# 1. How to optimize page load, rendering, responsiveness

[[5. FED/3. JavaScript/5. Performance/3. critical rendering path]] · [[5. FED/3. JavaScript/5. Performance/1. Debouncing, throttling]] · [[5. FED/2. CSS/7. Performance optimization]]

## Метрики

| Метрика | Что измеряет |
| :-- | :-- |
| **FCP** (First Contentful Paint) | Первые пиксели на экране |
| **LCP** (Largest Contentful Paint) | Появление самого большого контента (изображение, текст) |
| **TTI** (Time to Interactive) | Страница готова к взаимодействию |
| **INP** (Interaction to Next Paint) | Задержка от клика до отрисовки |
| **CLS** (Cumulative Layout Shift) | Сдвиги контента при загрузке |

## Page load times

### 1. Критический путь рендеринга

Браузер не может отрисовать страницу без DOM и CSSOM. Блокирующие ресурсы откладывают FCP. [[5. FED/3. JavaScript/5. Performance/3. critical rendering path]]

#### CSS блокирует рендер

Каждый `<link rel="stylesheet">` останавливает построение render tree до загрузки и парсинга CSS.

```html
<!-- Плохо: 3 файла = 3 блока -->
<link rel="stylesheet" href="reset.css">
<link rel="stylesheet" href="components.css">
<link rel="stylesheet" href="main.css">

<!-- Лучше: объединить в один или критичный inline -->
<head>
  <style>
    /* Только стили для above-the-fold: header, hero, первый экран */
    .header { height: 60px; }
    .hero { font-size: 2rem; min-height: 400px; }
  </style>
  <link rel="stylesheet" href="main.css" media="print" onload="this.media='all'">
  <noscript><link rel="stylesheet" href="main.css"></noscript>
</head>
```

`media="print"` — браузер не блокирует рендер. `onload="this.media='all'"` — после загрузки применяет к экрану.

#### JavaScript блокирует парсинг HTML

Скрипт без атрибутов останавливает парсинг, выполняется до продолжения.

```html
<!-- Плохо: парсинг ждёт загрузки и выполнения app.js -->
<script src="app.js"></script>

<!-- defer: загружается параллельно, выполняется после DOM -->
<script src="app.js" defer></script>

<!-- async: выполнится когда загрузится, порядок не гарантирован -->
<script src="analytics.js" async></script>
```

**defer** — порядок сохранён, выполнение после DOMContentLoaded. **async** — для независимых скриптов (аналитика, виджеты).

```html
<!-- Скрипты в конце body — не блокируют парсинг контента -->
<body>
  <!-- контент -->
  <script src="app.js"></script>
</body>
```

### 2. Изображения

#### Lazy load

```html
<!-- Нативный: браузер сам решает когда загружать -->
<img src="photo.jpg" loading="lazy" alt="...">

<!-- С fallback для старых браузеров -->
<img src="placeholder.jpg" data-src="large.jpg" loading="lazy" alt="...">
<script>
  document.querySelectorAll('img[data-src]').forEach(img => {
    img.src = img.dataset.src;
  });
</script>
```

#### LCP — приоритет для hero-изображения

```html
<img src="hero.jpg" fetchpriority="high" alt="Главный баннер">
```

Если hero-изображение ниже — `loading="lazy"` не нужен, браузер загрузит сразу.

#### Responsive images

```html
<!-- Браузер выберет размер под viewport -->
<img srcset="
  small.jpg 480w,
  medium.jpg 800w,
  large.jpg 1200w
" sizes="(max-width: 600px) 100vw, 50vw" src="fallback.jpg" alt="...">
```

`sizes` — размер изображения в layout. `srcset` — варианты под разные ширины.

### 3. Preload, prefetch

```html
<!-- Preload: критичный ресурс, загрузить сейчас -->
<link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin>
<link rel="preload" href="hero.jpg" as="image">

<!-- Prefetch: следующий экран, низкий приоритет -->
<link rel="prefetch" href="/dashboard.js">
```

`preload` — высокий приоритет. `prefetch` — фоновый. Шрифты — `crossorigin` обязателен.

### 4. Code splitting

```javascript
// React: lazy + Suspense
const Dashboard = lazy(() => import('./Dashboard'));
const Settings = lazy(() => import('./Settings'));

<Suspense fallback={<Spinner />}>
  <Routes>
    <Route path="/dashboard" element={<Dashboard />} />
    <Route path="/settings" element={<Settings />} />
  </Routes>
</Suspense>
```

Каждый route — отдельный chunk. Загружается при переходе.

### 5. Минификация и сжатие

- **Minify** — убрать пробелы, комментарии, сократить имена
- **Gzip/Brotli** — сервер отдаёт сжатый контент
- **Tree shaking** — удалить неиспользуемый код (ES modules)

## Rendering

### 1. Reflow и layout thrashing

Браузер оптимизирует batch: читает все layout-свойства, потом пересчитывает. Чередование read-write заставляет пересчитывать многократно.

#### Свойства, вызывающие reflow

`offsetWidth`, `offsetHeight`, `clientWidth`, `clientHeight`, `scrollTop`, `scrollLeft`, `getBoundingClientRect()`, `getComputedStyle()` — при чтении браузер пересчитывает layout.

```javascript
// Плохо: на каждой итерации — layout → reflow
for (let i = 0; i < elements.length; i++) {
  elements[i].style.width = elements[i].offsetWidth + 10 + 'px';
}

// Хорошо: batch read, batch write
const widths = elements.map(el => el.offsetWidth);
elements.forEach((el, i) => {
  el.style.width = widths[i] + 10 + 'px';
});
```

#### DocumentFragment для массовой вставки

```javascript
const fragment = document.createDocumentFragment();
items.forEach(item => {
  const li = document.createElement('li');
  li.textContent = item;
  fragment.appendChild(li);
});
list.appendChild(fragment);  // один reflow вместо N
```

### 2. Paint и compositing

**Paint** — отрисовка пикселей. **Composite** — объединение слоёв. `transform` и `opacity` не вызывают layout и paint — только compositing.

```css
/* Плохо: меняет layout → reflow → repaint */
.modal { left: 50%; top: 50%; }
.slide { width: 100px; }

/* Хорошо: только compositing */
.modal { transform: translate(-50%, -50%); }
.slide { transform: scaleX(1); }
```

Свойства только для compositing: `transform`, `opacity`, `filter` (в некоторых случаях).

### 3. will-change

Подсказка браузеру о будущих изменениях. Использовать точечно.

```css
/* Перед анимацией */
.slider:hover { will-change: transform; }

/* После анимации — снять */
.slider { will-change: auto; }
```

Постоянный `will-change` на многих элементах — лишние слои, больше памяти.

### 4. contain

Изоляция поддерева: изменения не влияют на остальную страницу.

```css
.card {
  contain: layout;   /* layout внутри карточки */
  contain: paint;    /* paint не выходит за границы */
  contain: style;    /* счётчики, quotes изолированы */
  contain: strict;   /* layout + paint + style */
}
```

Полезно для карточек, виджетов, списков.

### 5. Виртуализация списков

Рендерить только видимые элементы. Для 10000 элементов — 20–30 в DOM.

```javascript
const observer = new IntersectionObserver(
  (entries) => {
    entries.forEach((entry) => {
      if (entry.isIntersecting) {
        const index = +entry.target.dataset.index;
        const item = items[index];
        entry.target.innerHTML = renderItem(item);
        observer.unobserve(entry.target);
      }
    });
  },
  { rootMargin: '100px' }
);

// Создаём placeholder-элементы по высоте
items.forEach((_, i) => {
  const div = document.createElement('div');
  div.dataset.index = i;
  div.style.height = '50px';
  div.style.minHeight = '50px';
  list.appendChild(div);
  observer.observe(div);
});
```

Библиотеки: react-window, react-virtualized, virtua.

### 6. Селекторы

Глубокие и сложные селекторы медленнее. [[5. FED/2. CSS/7. Performance optimization]]

```css
/* Медленнее: много совпадений */
div ul li a span { }

/* Быстрее: класс */
.nav-link { }
```

## Responsiveness

### 1. Main thread

JavaScript, layout, paint — всё на одном потоке. Долгая задача (>50 мс) блокирует ввод, скролл, анимации.

### 2. Web Workers

Тяжёлые вычисления (парсинг, сортировка, криптография) — в отдельный поток.

```javascript
// main.js
const worker = new Worker('compute.js');
worker.postMessage({ data: largeArray });
worker.onmessage = (e) => {
  updateUI(e.data);
};

// compute.js
self.onmessage = (e) => {
  const result = e.data.data.sort((a, b) => a - b);
  self.postMessage(result);
};
```

Ограничения: нет доступа к DOM, `window`. Обмен только через `postMessage` (копирование данных).

### 3. Разбиение long tasks

Задачи >50 мс разбивать на chunks, чтобы браузер успевал обработать ввод.

```javascript
// Плохо: 100 мс блокирует UI
items.forEach(heavyProcess);

// Хорошо: yield каждые 50 элементов
async function processChunked(items, chunkSize = 50) {
  for (let i = 0; i < items.length; i += chunkSize) {
    const chunk = items.slice(i, i + chunkSize);
    chunk.forEach(heavyProcess);
    await new Promise(r => setTimeout(r, 0));  // yield
  }
}
```

`setTimeout(0)` — отдаёт управление event loop, следующий chunk выполнится после обработки других задач.

### 4. Debounce и Throttle

[[5. FED/3. JavaScript/5. Performance/1. Debouncing, throttling]]

```javascript
// Input, resize: debounce — выполнить после паузы
const handleSearch = debounce((q) => fetch(`/api?q=${q}`), 300);
input.addEventListener('input', (e) => handleSearch(e.target.value));

// Scroll, mousemove: throttle — не чаще раза в N мс
const handleScroll = throttle(() => updatePosition(), 100);
window.addEventListener('scroll', handleScroll);
```

### 5. Passive event listeners

По умолчанию браузер ждёт `preventDefault()` в обработчике scroll. С `passive: true` — не ждёт, скролл плавнее.

```javascript
element.addEventListener('scroll', handler, { passive: true });
```

Нельзя вызывать `preventDefault()` в passive-обработчике.

### 6. requestAnimationFrame

Синхронизация с частотой обновления экрана (~60 fps). Для анимаций и визуальных обновлений.

```javascript
let ticking = false;
function onScroll() {
  if (!ticking) {
    requestAnimationFrame(() => {
      updateParallax();
      ticking = false;
    });
    ticking = true;
  }
}
window.addEventListener('scroll', onScroll);
```

### 7. Scheduler.yield (эксперимент)

Chrome: `scheduler.yield()` — отдать управление с приоритетом.

```javascript
if ('scheduler' in globalThis && 'yield' in scheduler) {
  await scheduler.yield();
}
```
