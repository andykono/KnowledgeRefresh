# 2. virtual DOM

[[5. FED/3. JavaScript/5. Performance/1. Debouncing, throttling]] · [[5. FED/3. JavaScript/5. Performance/3. critical rendering path]]

## Зачем нужен (в контексте фреймворков)

**Virtual DOM** — легковесное представление реального DOM в виде JS-объектов. React, Vue и др. обновляют его при каждом рендере, затем сравнивают с предыдущей версией (diff) и вносят в DOM только нужные изменения. Цель — меньше прямых манипуляций с DOM и более предсказуемая производительность.

## Как это работает

1. **Рендер** — компонент возвращает описание UI (JSX → объекты)
2. **Virtual tree** — дерево объектов `{ type, props, children }`
3. **Diff** — сравнение нового дерева с предыдущим
4. **Patch** — применение минимального набора изменений к реальному DOM

## Пример (упрощённо)

```javascript
// Виртуальный узел
const vNode = {
  type: 'div',
  props: { className: 'card' },
  children: [
    { type: 'h2', props: {}, children: ['Title'] },
    { type: 'p', props: {}, children: ['Content'] }
  ]
};
```

Фреймворк сравнивает старый и новый vNode, находит отличия и обновляет только изменённые DOM-узлы.

## Плюсы

- Декларативный API — описываешь желаемое состояние, а не шаги
- Батчинг обновлений — несколько изменений состояния могут дать один patch
- Абстракция от платформы — тот же подход для web, mobile (React Native)

## Ограничения

- Накладные расходы на diff — для очень простых обновлений прямой DOM может быть быстрее
- Virtual DOM не «магически» ускоряет всё — он оптимизирует типичные сценарии (много мелких изменений)
