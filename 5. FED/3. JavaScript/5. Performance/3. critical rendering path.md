# 3. critical rendering path

[[5. FED/3. JavaScript/5. Performance/1. Debouncing, throttling]] · [[5. FED/3. JavaScript/5. Performance/2. virtual DOM]] · [[5. FED/2. CSS/7. Performance optimization]]

## Зачем понимать

**Critical Rendering Path (CRP)** — последовательность шагов, которые браузер выполняет, чтобы превратить HTML, CSS и JS в пиксели на экране. Оптимизация CRP — быстрее First Contentful Paint (FCP), меньше времени до интерактивности.

## Этапы CRP

1. **DOM** — парсинг HTML → DOM-дерево
2. **CSSOM** — парсинг CSS → CSS Object Model
3. **Render Tree** — объединение DOM + CSSOM (только видимые узлы)
4. **Layout** — расчёт позиций и размеров (reflow)
5. **Paint** — отрисовка пикселей
6. **Composite** — слои в финальное изображение

## Блокирующие ресурсы

- **CSS** — блокирует рендер. Браузер ждёт CSSOM, чтобы построить render tree.
- **JavaScript** — блокирует парсинг HTML (по умолчанию). Может блокировать и CSS.

### Пример: порядок загрузки

```html
<head>
  <link rel="stylesheet" href="styles.css">  <!-- блокирует рендер -->
  <script src="app.js"></script>             <!-- блокирует парсинг, выполняется до продолжения -->
</head>
```

## Оптимизации

### 1. Критический CSS inline

Стили для above-the-fold контента — в `<head>`, остальное асинхронно.

```html
<head>
  <style>
    /* критичные стили для первого экрана */
    .hero { font-size: 2rem; }
  </style>
  <link rel="stylesheet" href="main.css" media="print" onload="this.media='all'">
</head>
```

### 2. Async/defer для скриптов

```html
<script src="app.js" defer></script>   <!-- после DOM, порядок сохранён -->
<script src="analytics.js" async></script>  <!-- выполнится когда загрузится, порядок не гарантирован -->
```

### 3. Отложенная загрузка некритичного CSS

```html
<link rel="preload" href="fonts.css" as="style" onload="this.onload=null;this.rel='stylesheet'">
```

### 4. Preload ключевых ресурсов

```html
<link rel="preload" href="main.js" as="script">
<link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin>
```

## Для чего это нужно

- **Быстрый FCP** — пользователь видит контент раньше
- **Меньше блокировок** — JS не останавливает парсинг, CSS загружается эффективнее
- **LCP (Largest Contentful Paint)** — критичный контент (изображение, текст) рендерится быстрее

## Пример: типичные проблемы

| Проблема | Решение |
| :-- | :-- |
| Большой CSS блокирует рендер | Inline критичного, async остального |
| JS блокирует парсинг | `defer` или `async` |
| Шрифты блокируют текст | `font-display: swap`, preload |
| Изображения выше сгиба не загружаются первыми | `fetchpriority="high"`, preload |
