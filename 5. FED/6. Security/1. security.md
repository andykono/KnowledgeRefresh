# 1. security

## XSS (Cross-Site Scripting)

**Суть:** внедрение вредоносного скрипта в страницу. Выполняется в контексте пользователя — доступ к cookies, localStorage, DOM, сессии.

### Типы

| Тип | Где внедряется |
| :-- | :-- |
| **Stored** | Данные с сервера (комментарии, профиль) |
| **Reflected** | URL-параметры, отражённые в ответе |
| **DOM-based** | Уязвимый код на клиенте, без участия сервера |

### Пример атаки

```javascript
// Плохо: пользовательский ввод напрямую в DOM
element.innerHTML = userInput;  // userInput = '<img src=x onerror="steal(document.cookie)">'

// Плохо: eval, innerHTML, document.write с данными
eval(userInput);
```

### Защита

- **Экранирование** — преобразовать `<`, `>`, `"`, `'`, `&` в HTML-сущности
- **Не использовать** `innerHTML`, `eval`, `document.write` с непроверенными данными
- **Текстовый контент** — `textContent`, не `innerHTML`

```javascript
// Хорошо
element.textContent = userInput;

// Если нужен HTML — санитизация (DOMPurify)
element.innerHTML = DOMPurify.sanitize(userInput);
```

- **Content Security Policy (CSP)** — ограничить источники скриптов
- **HttpOnly cookies** — JS не читает cookie сессии

## CSRF (Cross-Site Request Forgery)

**Суть:** запрос с чужого сайта от имени авторизованного пользователя. Браузер автоматически отправляет cookies на тот же origin.

### Пример атаки

```html
<!-- Злонамеренный сайт evil.com -->
<img src="https://bank.com/transfer?to=attacker&amount=1000">
<!-- Или форма с auto-submit -->
<form action="https://bank.com/transfer" method="POST">
  <input name="to" value="attacker">
  <input name="amount" value="1000">
  <script>document.forms[0].submit()</script>
</form>
```

Пользователь залогинен на bank.com — запрос уйдёт с его cookies.

### Защита

- **CSRF-токен** — случайное значение в форме, проверяется на сервере
- **SameSite cookie** — `SameSite=Strict` или `Lax` (не отправлять при cross-site запросах)
- **Проверка Origin/Referer** — сервер отклоняет запросы с чужих доменов
- **Дополнительная аутентификация** — повторный пароль для критичных действий

```javascript
// Токен в форме (сервер генерирует)
<input type="hidden" name="csrf_token" value="{{ csrfToken }}">
```

## CORS (Cross-Origin Resource Sharing)

**Суть:** механизм браузера, разрешающий запросы на другой origin. По умолчанию браузер блокирует cross-origin запросы (кроме простых: GET, форма, img).

### Как работает

1. Браузер отправляет **preflight** (OPTIONS) для «непростых» запросов
2. Сервер отвечает заголовками: `Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers`
3. Браузер разрешает или блокирует запрос

### Заголовки сервера

| Заголовок | Назначение |
| :-- | :-- |
| `Access-Control-Allow-Origin` | Разрешённые origins (`*` или конкретный домен) |
| `Access-Control-Allow-Methods` | GET, POST, PUT, DELETE и др. |
| `Access-Control-Allow-Headers` | Разрешённые заголовки запроса |
| `Access-Control-Allow-Credentials` | Разрешить cookies в cross-origin запросах |
| `Access-Control-Max-Age` | Кэш preflight в секундах |

### Пример

```javascript
// Запрос на api.example.com с app.example.com
fetch('https://api.example.com/data', {
  credentials: 'include'  // отправить cookies
});
// Сервер должен: Access-Control-Allow-Origin: https://app.example.com
// (не *) и Access-Control-Allow-Credentials: true
```

### Важно

- CORS — защита браузера, не сервера. Postman/curl обходят CORS
- `*` не совместим с `credentials: 'include'`
- Preflight не отправляется для «простых» запросов (GET, POST с form-urlencoded, ограниченные заголовки)

## Secure coding practices

### 1. Валидация на клиенте и сервере

Клиент — UX. Сервер — единственный источник истины. Не доверять данным с клиента.

### 2. Хранение чувствительных данных

- **Не хранить** пароли, токены в localStorage — доступны JS
- **HttpOnly cookies** для сессий — скрипт не читает
- **Secure** — cookie только по HTTPS
- **SameSite** — защита от CSRF

### 3. Зависимости

- Регулярно обновлять пакеты (`npm audit`)
- Проверять репутацию и лицензии
- Минимизировать зависимости

### 4. Ошибки и логирование

- Не показывать стек-трейсы и внутренние детали пользователю
- Не логировать пароли, токены, PII

### 5. HTTPS

- Все запросы по HTTPS
- HSTS — принудительный HTTPS

### 6. Content Security Policy

```http
Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self'
```

Ограничивает источники скриптов, стилей, изображений. Снижает риск XSS.

### 7. Subresource Integrity (SRI)

```html
<script src="https://cdn.example.com/lib.js"
  integrity="sha384-..."
  crossorigin="anonymous"></script>
```

Проверка целостности внешних скриптов — защита при компрометации CDN.
